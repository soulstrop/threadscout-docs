ADR 0011: Simplifying OIDC Handshake by Disabling PKCE

Status

Accepted

Context

During the transition to a SaaS-capable architecture with PocketID as the primary identity provider, we encountered a technical failure in the OIDC handshake. Specifically, when PKCE (Proof Key for Code Exchange) was enabled alongside a Client Secret, PocketID (v2.3.0) returned a 400 Bad Request with the error Invalid code verifier.

This occurred despite the frontend (NextAuth.js) correctly sending both the verifier and the secret. The conflict appears to be related to how the OIDC provider handles hybrid state across subdomain boundaries or a strict validation rule within PocketID that rejects redundant verification methods.

Decision

We will disable PKCE for the threadscout OIDC client in the PocketID configuration and rely strictly on the Client Secret for the authorization code exchange.

Rationale

  1. Confidential Client Model: ThreadScout is a confidential client because it has a secure backend (FastAPI) capable of storing and protecting a secret. In this model, the Client Secret provides sufficient proof of identity for the token exchange.
  2. Redundancy: While PKCE is highly recommended for Public clients (like single-page apps without a backend), its role in Confidential clients is to prevent authorization code injection. Since we are using HTTPS throughout the entire pipeline (Cloudflare/Caddy), the risk of code interception is mitigated.
  3. Compatibility: PocketID 2.3.0 demonstrated a clear inability to process the dual verification (Secret + PKCE) consistently. Disabling PKCE resolves the “Invalid code verifier” error and allows the login flow to complete.
  4. Operational Simplicity: Reducing the number of moving parts in the security handshake makes the system easier to debug and maintain across different environment profiles (Local, Hybrid, Live).

Consequences

  • Security Configuration: The PocketID dashboard must have the PKCE toggle set to OFF for the threadscout client.
  • NextAuth Compatibility: No changes are required to the NextAuth.js configuration, as it gracefully falls back to a secret-only exchange when the provider does not mandate PKCE.
  • Documentation: The PocketID Identity Flow Runbook must be updated to reflect this requirement.
