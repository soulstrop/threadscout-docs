Analysis: Tailscale HTTPS Investigation (Tabling)

Status: Tabled (March 5, 2026)

We attempted to enable natively trusted HTTPS for the ThreadScout stack across a Tailnet to support mobile testing (Pixel 6 Pro).

Approaches Tried

  1. Direct tls tailscale in Caddy: Failed because the standard Caddy binary lacks the caddy-tailscale module.
  2. Manual Tailscale Certs (tailscale cert):
    • Successfully generated for the base machine name (gmktec.tailab329.ts.net).
    • Failed for subdomains (auth.gmktec...) as Tailscale only issues certs for the exact machine DNS name.
  3. Port-Based Routing with Manual Certs:
    • Configured ThreadScout on :3443 and PocketID on :8443 using the same machine cert.
    • Encountered permission denied when Caddy (running as user) tried to read root-owned Tailscale keys.
    • Resolved permissions by copying certs to .certs/, but encountered persistent ERR_SSL_PROTOCOL_ERROR on mobile devices.
  4. Internal TLS with On-Demand:
    • Caddy successfully generated internal certs for the Tailnet hostname.
    • Handshake failed on mobile because the mobile browser does not trust Caddy’s internal Root CA.

Root Cause of SSL Failures

Mobile browsers (Chrome on Android) are extremely strict about SSL handshakes on custom ports. Without a natively trusted CA (like Tailscale’s or Let’s Encrypt), the handshake is often rejected before the “Proceed Unsafely” option can even be presented.

Future Ideas

  • xcaddy: Build a custom Caddy binary with the tailscale module to allow Caddy to fetch its own certs via the Tailscale socket.
  • Tailscale Serve: Use tailscale serve to proxy traffic from the tailnet directly to the local ports, letting Tailscale handle the TLS termination.
  • HTTP-Only Circle of Trust: Treat the Tailnet as a secure internal network and use plain HTTP for rapid UI testing.

Firewall Changes Made

  • sudo ufw allow in on tailscale0
  • sudo ufw allow in on tailscale0 to any port 3443
