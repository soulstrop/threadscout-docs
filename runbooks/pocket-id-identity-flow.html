Pocket Id Identity Flow

PocketID Identity Flow & Troubleshooting

See also: Runbook: Self-Hosting PocketID for initial installation and container configuration.

This runbook documents how ThreadScout integrates with PocketID as its primary identity provider. It covers the setup, authorization flows across different environments, and common troubleshooting steps.

1. Environment Configurations

ThreadScout uses three primary environment profiles, each with a different networking strategy for PocketID.

Profile Public URL (Issuer) Internal URL (JWKS) Use Case
Local https://auth.localhost:3443 http://localhost:8081 Pure local development on host.
Hybrid https://auth.constans.dev http://localhost:8081 Remote testing via Cloudflare Tunnel.
Live https://auth.threadscout.constans.dev http://pocket-id:8081 Full production-like Docker setup.

2. Setup Flow: Client Registration

Before users can log in, ThreadScout must be registered as an OIDC Client in PocketID.

A. Required Client Settings

In the PocketID Admin Panel (Administration -> OIDC Clients):

  • Client ID: threadscout
  • Name: ThreadScout
  • Public Client: OFF (Confidential client flow)
  • PKCE: OFF (See ADR 0011)
  • Redirect URIs:
    • https://threadscout.constans.dev/api/auth/callback/pocketid
    • https://localhost:3443/api/auth/callback/pocketid
  • Logout Callback URIs:
    • https://threadscout.constans.dev
    • https://localhost:3443

B. Access Control (Groups)

PocketID defaults to “Deny All” for new clients. You must permit a group to access the client:

  1. Create a User Group (e.g., resellers).
  2. Add your user to this group.
  3. In the threadscout client settings, add this group to “Allowed User Groups”.

3. Infrastructure Requirements (Cloudflare & Proxy)

A. Cloudflare Subdomain & Rocket Loader

If using Cloudflare for public ingress:

  1. Public Hostname: Add auth.yourdomain.com to your Tunnel configuration.
  2. Rocket Loader: MUST BE OFF for the auth subdomain. Rocket Loader injects scripts that violate PocketID’s strict Content Security Policy (CSP).

B. Caddy Header Integrity

Caddy must tell the applications that the public connection is secure, even if the internal link is HTTP.

  • Header: X-Forwarded-Proto: https
  • Why: NextAuth will ignore session cookies if it detects an insecure (http) scheme.

4. Token & Session Management

ThreadScout uses a specialized session strategy to bridge the Frontend and Backend:

  1. ID Token vs. Access Token: The Frontend (NextAuth) is configured to send the ID Token to the Backend.
    • Rationale: The ID Token contains user profile claims (like email) which are required by the Backend for JIT user provisioning. The standard Access Token in PocketID 2.3.0 does not include these claims.
  2. Session Cookies:
    • In production/hybrid modes, cookies use the __Secure- prefix.
    • Cookies are scoped to the parent domain (e.g., .constans.dev) to ensure compatibility across the app and auth subdomains.

6. Troubleshooting “The Session Bounce”

If you are redirected back to the login screen immediately after a “successful” PocketID login:

  1. Clear Browser Data: Use Chrome DevTools -> Application -> Clear site data. Old cookies or stale state are the #1 cause of bounces.
  2. Disable Adblockers: Adblockers often block /api/auth/* requests.
  3. Check Backend Logs:
    • 400 Bad Request: Usually means the token is missing the email claim (check Token Selection).
    • 401 Unauthorized: Usually means X-Forwarded-Proto is incorrect or the signing keys (JWKS) are unreachable.
  4. Verify Issuer Match: The iss claim in the JWT MUST match THREADSCOUT_POCKET_ID_PUBLIC_URL exactly (including https://).
