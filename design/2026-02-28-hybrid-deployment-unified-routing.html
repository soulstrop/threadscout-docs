Technical Design Spec: Hybrid Deployment & Unified Domain Routing

This spec defines the architecture for ThreadScout’s Hybrid Deployment, enabling a unified user experience across Static content (GitHub Pages) and Dynamic application (Cloudflare Tunnel) using a single domain: threadscout.constans.dev.

1. Traceability

  • Goal: Unify the developer/user entry point and resolve “Subpath Proxy” routing issues.
  • Journey: [UJ-00: SaaS Auth], [UJ-00b: Beta Onboarding].
  • Phase: Phase 8: Production Readiness.

2. Participants & Responsibilities

Participant Domain/Hostname Responsibility
User threadscout.constans.dev The end-user browser interacting with the application.
Cloudflare Worker Edge Interceptor “Traffic Cop” routing traffic based on path and session state.
GitHub Pages soulstrop.github.io/threadscout-docs/ Serves Static Landing Page, Documentation, and Legal pages.
Cloudflare Tunnel origin-threadscout.constans.dev Secure bridge to the Dynamic Application (Frontend/Backend).
Origin App localhost:8080 The core ThreadScout stack (Next.js + FastAPI) running on the deployment platform.

3. Deployment Distribution

A. Static Assets (GitHub Pages)

These documents/pages are exported as static HTML and deployed to GitHub Pages for high availability and zero-cost hosting of public-facing content.

  • Landing Page: The public entry point (/).
  • Documentation: All developer and user documentation (/docs/*).
  • Legal: Privacy Policy and Terms of Service (if static).
  • Waitlist: The static portion of the beta waitlist UI.

B. Dynamic Application (ThreadScout Platform)

The core application logic and authenticated experience run on the chosen deployment platform (e.g., K8s, VPS, or local host via Tunnel).

  • Authentication: /login, /api/auth/*.
  • Dashboard: The main reseller interface (/dashboard, /items).
  • API: The FastAPI backend (/api/*).
  • Admin: System administration and settings.

4. Unified Routing Logic (Cloudflare Worker)

The Worker acts as a reverse proxy, deciding where to send each request.

A. Routing Matrix

Path Pattern Condition Target
/api/*, /_next/*, /login, /auth/* (Any) Dynamic Origin
/dashboard/*, /items/*, /admin/* (Any) Dynamic Origin
/docs/* (Any) GitHub Pages
/ (Root) next-auth.session-token exists Dynamic Origin (Authenticated Home)
/ (Root) No session cookie GitHub Pages (Public Landing Page)
/* (Fallback) (Default) GitHub Pages

B. Fallback Strategy

  • Origin Down: If the Dynamic Origin returns a 503 or fails to respond, the Worker returns a custom “System Maintenance” page instead of a generic Cloudflare error.
  • Static Down: If GitHub Pages is unreachable, the Worker attempts to serve a cached version of the landing page or redirects to a status page.

6. Implementation Details

  • Host Header Rewriting: Requests to GitHub Pages MUST have the Host header rewritten to soulstrop.github.io to be accepted.
  • Cookie Scope: Cookies must be scoped to threadscout.constans.dev to ensure they are visible to the Worker for routing decisions.
  • Path Normalization: The Worker must handle trailing slashes for /docs to prevent broken relative links in the static documentation.
