ThreadScout Developer Guide

This guide defines the standard workflow and tooling for the ThreadScout engineering squad. Our goal is a Zero-Friction environment where any developer can go from git clone to a verified local environment in under 5 minutes.

1. Environment & Toolchain

ThreadScout uses mise as the single source of truth for the project’s toolchain and environment variables.

1.1. Core Tools

  • Python (uv): High-performance package management and virtual environment isolation for the backend.
  • Node.js (npm): Frontend development and Playwright E2E testing.
  • Caddy: Local reverse proxy for HTTPS termination and API routing.
  • fnox: Encrypted secret management.
  • Cloudflared: Public ingress tunnel for testing eBay OAuth callbacks locally.

1.2. Setup

# Clone the repository
git clone <repo-url> && cd threadscout

# Install all tools and project dependencies
mise run setup    # One-time system setup (browsers, etc.)
mise run install  # Frontend & Backend dependencies

1.3. Environment Profiles

ThreadScout uses MISE_ENV to switch between standardized development profiles. Each profile has a corresponding mise.<profile>.toml file that defines its specific behavior.

  • local: Default for rapid iteration. Uses Mock Auth and Mock eBay. Ingress: localhost:3443.
  • hybrid: Real Auth (PocketID), Mock eBay. Ingress: threadscout.constans.dev.
  • public-mock: Mock Auth, Mock eBay. Ingress: threadscout.constans.dev. Used for testing public-domain UX.
  • live: Real Auth, Real eBay. Ingress: threadscout.constans.dev. Handle with care.

1.4. Maintenance

  • mise run reset: Clears local storage and databases.
  • mise run reset --hard: Additionally clears all build caches (.next, .ruff_cache, .venv, etc.).

1.5. Version Management (mise.lock)

To ensure environment stability while maintaining flexibility, ThreadScout uses a Lock/Upgrade/Re-lock workflow. This allows us to keep tool definitions as latest while recording exact versions for reproducibility.

  • Locking (Record current versions): Run mise lock. This creates/updates mise.lock with the specific versions currently in use (e.g., uv@0.10.8).

  • Upgrading (Refresh versions): Run mise upgrade (or mise up). This ignores the existing lock, fetches the latest available tools, installs them, and automatically updates mise.lock.

  • Clean Slate (Emergency Unlock): If you encounter a backend conflict (e.g., Conda solve errors), use the following sequence:

    rm mise.lock && mise install && mise lock

3. Standard Developer Workflow

Our workflow is centered around the Unified Launch Task, which orchestrates all required services and sidecars based on your selected profile.

3.1. Mock Mode (Fast Iteration)

Use this for UI/UX work or logic changes that don’t require real AI or eBay APIs.

  • Command: mise run launch local
  • Effect:
    • MISE_ENV=local (Loads mise.local.toml)
    • AI Vision returns simulated garment data.
    • Auth is bypassed (Robotic Login enabled).
    • eBay API calls are simulated.

3.2. Integrated Mode (Hybrid)

Use this for testing real AI extraction, enrichment, and OIDC authentication via the local PocketID sidecar.

  • Command: mise run launch hybrid
  • Requirements:
    • Valid fnox credentials for Google Vertex AI and eBay Developer Portal.
    • PocketID client registration (See mise run pocket-id:setup).
    • A running Cloudflare tunnel for callbacks (Started automatically by the launch task).

3.3. Containerized Mode (Production Parity)

Use this to verify deployment readiness and cross-service orchestration.

  • Command: mise run docker:up
  • Verification: Run E2E tests against the Docker stack with mise run check:docker.

4. Quality Standards & Verification

Every PR must pass the mise run check suite, which executes:

Tool Phase Goal
Ruff Linting Consistent code style and formatting.
Mypy Type-checking Static type safety in the backend.
Radon Complexity Monitor cyclomatic complexity (Maintainability).
Pytest Unit/Logic 100% coverage on core service logic.
Playwright E2E Functional verification of User Journeys.

5. Secret Management (fnox)

Secrets are managed centrally in backend/fnox.toml. To add a new secret:

  1. Add to fnox: fnox set threadscout/category/key "value"
  2. Map in backend: Update src/threadscout/config.py using Pydantic Settings.
  3. Local Access: mise automatically injects fnox secrets as environment variables into all mise run tasks.

6. Observability & Forensic Tooling

ThreadScout implements an “Agent-First” observability stack designed for high-fidelity debugging.

6.1. Unified Log Probing

Instead of tailing raw files, use the prober to search for patterns across service boundaries.

# Search all logs for a specific incident timestamp
./scripts/probe-logs.sh all --grep "2026-03-03T10:15" --context 50

# Monitor errors in a specific service
./scripts/probe-logs.sh backend --level ERROR

6.2. Security Auditing

Verify that the log scrubber is correctly masking PII (emails, secrets) before logs leave the local environment.

./scripts/audit-logs.sh --all

7. Hardware Isolation (The Sandbox)

For dangerous or unverified AI processing tasks, use the isolated MicroVM.

  • Task: mise run isolate
  • Philosophy: This launches a QEMU MicroVM with AppArmor profiles and Virtio-9p file sharing, ensuring the host machine remains secure during unverified code execution.

7. Documentation Workflow

We treat documentation as code. All .md files in docs/ are automatically processed into HTML for the documentation hub.

  • Build: mise run docs:build
  • Watch: mise run docs:watch (Auto-rebuilds on file change)
  • Preview: https://localhost:3443/docs

8. Operational Awareness (Ops Pulse)

Developers are expected to monitor the Ops Pulse Dashboard in the application (/ops).

  • Incidents: Review high-fidelity debug snapshots from user reports.
  • Telemetry: Monitor the Prometheus endpoint (/api/metrics) for performance regressions.
  • Logs: Backend logs are structured JSON (Structlog) for easy parsing.
