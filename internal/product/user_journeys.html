ThreadScout User Journeys

This document defines the primary paths a user takes through the application. These journeys serve as the source of truth for our Playwright E2E tests.

[UJ-00] SaaS Authentication & Onboarding

Goal: Access the reseller dashboard securely via Passkey or Mock login.

  1. Navigation: User opens the app. If THREADSCOUT_ENABLE_AUTH is true, they are redirected to /login.
  2. Selection: User chooses their preferred login method (Pocket ID, Google, Apple, eBay, or Mock).
  3. Authentication: The identity provider verifies the user.
  4. Just-In-Time Provisioning: ThreadScout backend receives the token, verifies it, and automatically creates a User record if it’s the first time they’ve logged in.
  5. Completion: User is redirected to the root dashboard (/), ready for UJ-01.

[UJ-00b] Beta Onboarding & Growth

Goal: Convert invited beta testers and capture interest from uninvited visitors through a guided, frictionless experience built on the ThreadScout Unified Design System (TSDS).

  1. Landing:
    • Invited: Arrives via https://ts.constans.dev/?token=BETA_XYZ. The system recognizes the token and permits account creation.
    • Public: Arrives at the landing page. If no token is present, they see a “Join the Waitlist” call-to-action.
  2. Waitlist: Uninvited users enter their email to be notified when more beta slots open.
  3. Onboarding Tour: Upon first successful login, the user is presented with a “swipeable” 3-step tutorial:
    • Slide 1: “Dashing Analysis” - Intro to AI Vision.
    • Slide 2: “Smart Research” - Intro to Enrichment & Comps.
    • Slide 3: “One-Click Drafts” - Intro to eBay Integration.
  4. UX Flexibility: Users can skip the tutorial immediately. A “Show Tutorial” link remains available in the Admin dashboard for later reference.
  5. Activation: User enters the IDLE state of the core reseller loop (UJ-01).

[UJ-01] The Core Reseller Loop

Goal: Take a raw set of photos and convert them into a structured, stored item record ready for listing.

  1. Discovery: User opens the app and sees a gallery of un-processed images (Discovery via Ingest Backend).
  2. Selection: User selects one or more images representing a single clothing item.
  3. Visual Analysis: User triggers AI Analysis. Gemini extracts Brand, Size, Material, and Condition.
  4. Verification: User reviews the “Listing Preview” to ensure the AI interpreted the photos correctly.
  5. Finalization: User clicks “Finish”. The images are moved to permanent storage/ and a database record is created.

[UJ-02] Iterative Market Research & Enrichment

Goal: Iteratively improve the listing accuracy and set a competitive price using external data.

  1. Phase 1: Visual Research (Images)

    • User clicks “Images” to find the item’s retail/catalog presence on the web.
    • User finds a match, pastes the URL, and triggers “Enrich”.
    • Result: The listing is enriched with high-fidelity “Identity” data (Model Name, MSRP, official colorways).

  2. Phase 2: Market Research (Sold)

    • User clicks “Sold”. This search now uses the refined keywords (e.g., specific Model Name) obtained from Phase 1.
    • User finds a comparable sold listing on eBay, pastes the URL, and triggers “Enrich” again.
    • Result: The listing is further enriched with “Market” data (technical aspects from other sellers, pricing context, and professional descriptions).

  3. Final Review: User reviews the cumulative result before finishing.

[UJ-03] eBay Integration Setup

Goal: Securely connect the local instance to the eBay API.

  1. Navigation: User navigates to the Admin page.
  2. Consent: User clicks “Connect eBay” and is redirected to the eBay OAuth consent page.
  3. Callback: After consenting, user is redirected back to ThreadScout.
  4. Credential Security: ThreadScout exchanges the code for tokens, encrypts them using Fernet (AES-128), and persists them to SQLite.
  5. Verification: Admin page now shows “Connected” status with an expiration date.

[UJ-04] System Administration

Goal: Monitor costs, configure application behavior, and customize the user experience.

  1. Usage Audit: User reviews the “AI Token Usage” card to see cumulative and 24-hour costs.
  2. Ingest Configuration: User switches between “Local Filesystem” and “S3-Compatible Storage”.
  3. Appearance: User toggles between Light, Dark, and System themes to suit their environment. Changes are persisted locally to ensure a consistent experience across sessions.
  4. Storage (BYOS): User configures their own S3-compatible bucket (e.g., Cloudflare R2) via the Admin dashboard. Credentials are encrypted at-rest using Fernet.

[UJ-05] Draft Synthesis & Push

Goal: Finalize listing metadata and publish it as a draft to the connected eBay account.

  1. Synthesis: User enters the “Synthesis Editor” for a stored item. They review the AI-generated Title and Description, making manual adjustments as needed.
  2. Market Pricing: User sets the final price based on the research conducted in UJ-02.
  3. Policy Selection: User selects from their fetched eBay Shipping, Payment, and Return policies.
  4. Push: User clicks “Push to eBay”. The system sequentially calls the eBay Inventory API to create an item and the Offer API to create a draft listing.
  5. Completion: A direct link to the eBay draft is provided for final review and publishing.

[UJ-06] Mobile Share-to-Ingest

Goal: Seamlessly transfer photos from a mobile device’s native gallery to the ThreadScout ingest queue.

  1. Selection: User is using their phone to take product photos. They select multiple photos in their native gallery app (e.g., iOS Photos or Google Photos).
  2. Share Interaction: User clicks the native system “Share” button.
  3. Target Selection: User selects ThreadScout from the share sheet.
    • Android: Handled via PWA Web Share Target API.
    • iOS: Handled via a custom “Share to ThreadScout” iOS Shortcut.
  4. Transfer: The photos are POSTed to the ThreadScout /api/ingest/upload endpoint.
  5. Verification: User opens the ThreadScout web app and sees the newly shared photos at the top of their Ingest Gallery, ready for UJ-01.

[UJ-07] Billing & Subscription Management

Goal: Manage user tiers (Free vs. Pro) and enforce corresponding AI token and listing quotas using Stripe.

  1. Upgrade Path: User navigates to the Admin panel and sees their current usage quotas. They click “Upgrade to Pro.”
  2. Checkout: User is redirected to a secure Stripe Checkout page to enter payment details.
  3. Tier Activation: Upon successful payment, a Stripe Webhook notifies the ThreadScout backend to update the user’s tier to PRO.
  4. Enforcement: The backend immediately reflects the new quotas for AI analysis and enables “Pro-only” features like BYOS (S3).
  5. Self-Service: User can return to the Admin panel to launch the Stripe Customer Portal for managing their subscription (upgrading, downgrading, or canceling).

[UJ-08] Operations & Incident Triage

Goal: Provide a highly integrated, AI-assisted workflow for incident reporting, triage, and resolution, bridging the gap between end users, operations managers, and the development squad. Built on the professional high-contrast aesthetic of TSDS.

  1. Reporting & Instant Feedback:
    • User encounters a problem and submits a report via the UI.
    • System automatically enriches the report with local state, logs, spans, and database data in isolation.
    • If the issue matches a known pattern, the system immediately presents the user with a temporary workaround or prompts for specific missing details.
    • Users can track the status of their reports in a “My Reports” portal.
  2. AI Enrichment & Triage:
    • An LLM with context of past issues and the codebase enriches the new report with assigned severity, categorization, potential root causes, and estimated blast radius.
    • The report is placed in the Triaged queue.
    • Ops users are signaled via the Ops Dashboard.
  3. Ops Management:
    • Ops user navigates to the Ops Dashboard and reviews highlighted issues lacking input.
    • Ops can view semantic clusters of bugs to trace quality drops to specific codebase areas.
    • Ops can override default AI priorities, mark issues as duplicates (linking to older issues), and utilize one-click feature toggles to disable broken components globally.
  4. Squad Resolution (TDD & Automation):
    • The AI Development Squad (Architect, Builder, Sentinel, Historian) accesses the prioritized queue.
    • Historian includes the issue in the stand-up status report.
    • Architect examines the issue data, proposes a root cause, and generates a failing automated test (Playwright/Pytest) acting as a quality gate.
    • Builder utilizes mise run repro <ISSUE_ID> to replicate the exact failure state in a sandbox and implements the fix to pass the test.
    • Sentinel validates the fix against the quality gate.
  5. Closure & Notification:
    • Upon Sentinel approval, the database is updated, the issue is closed (indexed by date with full history for regression tracking).
    • Historian updates the squad state.
    • The user is notified of the fix (if contact info was provided) and the Ops Dashboard reflects the recently completed fix.

[UJ-09] Account Recovery & Identity Linking (SPEC-40)

Goal: Ensure continuous access to the ThreadScout account by linking multiple social identities and configuring recovery methods.

  1. Identity Linking: An authenticated user navigates to the Admin dashboard. They click “Link Account” and choose an additional social identity provider (e.g., if they originally logged in with Google, they link their eBay account).
  2. Consolidation: The backend verifies the new identity and links it to the existing User record.
  3. Redundancy: User configures a recovery email in the dashboard.
  4. Recovery Access: In the event that their primary social account is inaccessible or the passkey is lost, the user can log in via any of their linked accounts or utilize the recovery email flow to regain access to their siloed storage and item history.
