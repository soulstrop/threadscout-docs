ThreadScout Deployment Guide

This document provides technical instructions for deploying, monitoring, and maintaining ThreadScout. For the architectural rationale, see ARCHITECTURE.

Phase 1: Preparation & Setup

1.1 Prerequisites

Before deployment, ensure the following are installed and configured:

  • mise: For managing all tools and tasks.
  • fnox: For managing encrypted secrets.
  • mise-env-fnox: (Plugin) Automatically injected into mise to bridge secrets into the environment.
  • Docker & Docker Compose: Required for containerized sidecars (PocketID).
  • Google Cloud Platform: Vertex AI API enabled and gcloud auth application-default login run.

1.2 Environment Configuration

ThreadScout uses Federated Orchestration (SPEC-61). Environment authority is centralized at the root mise.toml, which delegates execution to specialized submodules.

Environment Profiles

Configuration is driven by the FNOX_PROFILE variable (defaults to local). Each profile corresponds to a mise.<profile>.toml file:

  • local: (Default) Mock Auth, Mock AI, Mock eBay. Ingress: localhost:3443.
  • hybrid: Real Auth (PocketID), Real AI, Mock eBay. Ingress: threadscout.constans.dev.
  • live: Real everything. Ingress: threadscout.constans.dev. Production use only.
Ingress & Network Topology

ThreadScout uses an Ingress Authority model to ensure consistent behavior across deployment modes.

  • THREADSCOUT_INGRESS_MODE: REQUIRED.
    • DIRECT: No proxy. HTTP only. Mock auth permitted.
    • PROXY: Proxied traffic (Caddy). Strict local OIDC.
    • TUNNEL: Public traffic (Cloudflare). Strict secure cookies and public OIDC.
Core Security & Identity Variables

ThreadScout uses robust configuration templating (SPEC-47) to ensure safe environment variable injection without mise startup crashes.

  • THREADSCOUT_APP_ENV: mocking, development, or production.

  • THREADSCOUT_SECRET_KEY: CRITICAL. A unique, high-entropy string used for session signing.

  • THREADSCOUT_ENCRYPTION_KEY: A Fernet (AES-128) key for at-rest token protection.

  • THREADSCOUT_DEBUG: Must be false in production to prevent sensitive trace leakage.

Identity (OIDC / PocketID):

  • THREADSCOUT_ENABLE_AUTH: Must be true for public deployments.
  • THREADSCOUT_POCKET_ID_PUBLIC_URL: The public-facing URL of the auth service (e.g., https://auth.threadscout.com). This is the issuer claim in JWTs.
  • THREADSCOUT_POCKET_ID_INTERNAL_URL: (Optional) The internal network URL (e.g., http://pocket-id:8080). If set, the backend will fetch JWKS via this URL to bypass reverse proxies.
  • THREADSCOUT_OIDC_AUDIENCE: Defaults to threadscout.

1.3 Environment Variables Reference

ThreadScout uses strict environment variable naming. Backend configuration is managed via Pydantic Settings, while the Frontend uses a mix of Next.js public variables and secure server-side variables.

  • Core System Variables: THREADSCOUT_APP_ENV / NEXT_PUBLIC_APP_ENV, THREADSCOUT_INGRESS_MODE / NEXT_PUBLIC_THREADSCOUT_INGRESS_MODE, THREADSCOUT_DEBUG / NEXT_PUBLIC_DEBUG, THREADSCOUT_LOG_LEVEL / NEXT_PUBLIC_LOG_LEVEL, THREADSCOUT_DEEP_TRACE.
  • Security & Authentication: THREADSCOUT_SECRET_KEY, THREADSCOUT_ENCRYPTION_KEY, THREADSCOUT_NEXTAUTH_SECRET, THREADSCOUT_ENABLE_AUTH, THREADSCOUT_BYPASS_BETA_TOKEN (used to bypass the beta invitation wall for E2E testing or initial account setup in non-production environments).
  • Identity (OIDC): THREADSCOUT_POCKET_ID_PUBLIC_URL, THREADSCOUT_POCKET_ID_INTERNAL_URL, THREADSCOUT_OIDC_AUDIENCE, THREADSCOUT_POCKET_ID_CLIENT_ID, THREADSCOUT_POCKET_ID_CLIENT_SECRET, THREADSCOUT_GOOGLE_CLIENT_ID, THREADSCOUT_GOOGLE_CLIENT_SECRET, THREADSCOUT_EBAY_OIDC_CLIENT_ID, THREADSCOUT_EBAY_OIDC_CLIENT_SECRET.
  • Integrations:
    • eBay APIs: THREADSCOUT_EBAY_CLIENT_ID, THREADSCOUT_EBAY_CLIENT_SECRET, THREADSCOUT_EBAY_DEV_ID, THREADSCOUT_EBAY_RU_NAME, THREADSCOUT_EBAY_MARKETPLACE_ID, THREADSCOUT_EBAY_SANDBOX.
    • Google Cloud / Vertex AI: THREADSCOUT_GOOGLE_CLOUD_PROJECT, THREADSCOUT_GOOGLE_CLOUD_LOCATION, THREADSCOUT_MODEL_ANALYSIS, THREADSCOUT_MODEL_ENRICH.
    • Stripe Billing: THREADSCOUT_STRIPE_API_KEY, THREADSCOUT_STRIPE_WEBHOOK_SECRET, THREADSCOUT_STRIPE_PRO_PRICE_ID.
  • Observability & Telemetry: THREADSCOUT_SENTRY_DSN / NEXT_PUBLIC_SENTRY_DSN, THREADSCOUT_ENABLE_METRICS, THREADSCOUT_OPS_WEBHOOK_URL, THREADSCOUT_OPS_TOKEN.
  • Networking: THREADSCOUT_FRONTEND_URL, NEXT_PUBLIC_API_URL.

1.4 Secret Management (Federated fnox)

ThreadScout uses a Hierarchical Secret Architecture. Secrets are merged recursively:

  1. Root (/fnox.toml): Global infrastructure (Keys, OIDC Discovery).
  2. Backend (backend/fnox.toml): Python-specific (DB URLs, AI Aliases).
  3. Frontend (frontend/fnox.toml): Next.js-specific (API URLs, NextAuth Secrets).

Automatic Injection: The jdx/mise-env-fnox plugin ensures that running mise run <task> automatically loads the correct secrets for your current directory and profile. There is no need for fnox run wrappers.

Namespace Use Case pass Path Root
Development Local testing (local profile) threadscout/dev/
Production Live environments (hybrid, live) threadscout/prod/

Mandatory Production Keys:

  • threadscout/prod/encryption_key: A high-entropy Fernet key for database protection.
  • threadscout/prod/auth/secret_key: Unique string for backend session signing.
  • threadscout/prod/auth/nextauth_secret: Unique string for frontend authentication.
  • threadscout/prod/auth/pocket_id_client_secret: The OIDC client secret.

Phase 2: Starting the Stack

ThreadScout uses Caddy as a unified ingress for local HTTPS, required for eBay OAuth and secure cookie handling.

2.1 Standard Deployment (Mise)

Best for rapid development and host-native execution. The launch task orchestrates the full stack (Caddy, Backend, Frontend, Sidecars) based on your selected profile.

mise run setup                # Installs local CA, browsers, and sidecars
FNOX_PROFILE=hybrid mise run launch # Starts the integrated stack

2.2 Containerized Deployment (Docker)

Best for verified “Production-Ready” states.

docker compose up -d
# Access via Caddy at: https://threadscout.constans.dev

2.3 Hardware Isolation (MicroVM)

For high-security environments, isolate the backend in a QEMU MicroVM.

scripts/setup-isolation.sh
mise run proxy  # Run Caddy on host
mise run isolate # Run Backend in VM

2.4 Orchestration & Edge Fidelity (SPEC-45, SPEC-46)

ThreadScout uses automated readiness probes and local worker simulation to ensure parity between development and production.

  • Readiness Probes: The mise run launch task utilizes automated probes to ensure Caddy, the database, and the OIDC provider are responsive before marking the stack as healthy.
  • Local Worker Simulation: To simulate Cloudflare Worker behavior (e.g., origin spoofing and header injection), the local Caddy configuration mirrors the production edge logic, ensuring a 1:1 parity with the Cloudflare Ingress.

Phase 3: Monitoring & Observability

ThreadScout implements an “Agent-First” observability stack that persists logs even after process crashes.

3.1 Live Log Probing

Instead of tailing multiple windows, use the unified prober or the lnav-powered logs task.

# Monitor all logs in a unified, interactive terminal
mise run logs

# Monitor specific service errors
./scripts/probe-logs.sh backend --level ERROR

Search for specific event patterns across all services

./scripts/probe-logs.sh all –grep “OAuth Callback” –context 5

Check if the stack initialized correctly

./scripts/probe-logs.sh proxy –grep “serving”


### 3.2 Security Auditing
Before promoting to production, verify that the log scrubbing mechanism is active and that no PII has leaked.
```bash
# Audit all logs for unmasked emails or secrets
./scripts/audit-logs.sh --all

# Verify that PII is being redacted correctly
./scripts/audit-logs.sh backend --grep "REDACTED_EMAIL"

3.3 Health Checks & Dependency Probing

ThreadScout includes a deep health check endpoint for monitoring:

  • Endpoint: https://localhost:3443/api/health
  • Behavior: Returns 200 OK only if the database, Gemini AI, and (if enabled) the OIDC Identity Provider are responsive. Returns 503 Service Unavailable with a detailed JSON payload on failure.
  • Startup Probe: The backend performs an oidc_readiness_probe during initialization. If the OIDC provider is unreachable, it logs specific instructions to check sidecar status.

3.4 Metrics & Telemetry

  • Log Standardization: ThreadScout enforces UTC ISO 8601 timestamps across all logs. Ensure that your log aggregators are configured to expect this format.
  • Prometheus: Real-time API performance and AI token usage available at /api/metrics.
  • Sentry: Critical application failures and debug snapshots are automatically reported to the configured Sentry DSN.

3.5 Automated Ingress Cleanup (SPEC-41)

To maintain network hygiene and prevent stale configurations in hybrid environments, ThreadScout includes an automated ingress cleanup task.

  • Mechanism: The backend periodically audits active tunnels and local proxy rules.
  • Manual Trigger: Run mise run ingress:cleanup to forcefully prune expired ingress entries and sync the local Caddyfile with active Cloudflare tunnels.
  • Safety: The cleanup task preserves all entries marked with permanent: true in fnox-schema.json.

3.6 Ingress Repair & Connectivity Checks (SPEC-50)

If connectivity to the main app or OIDC service is interrupted, use the ingress repair tool.

  • Manual Trigger: Run mise run ingress:repair.
  • Behavior: Performs a multi-point reachability check for localhost:3000, localhost:8080, and the public tunnel URLs. If inconsistencies are found, it automatically regenerates the local Caddy configuration and restarts the proxy.

Phase 4: Crisis Response & Triage

When an incident is reported or a crash occurs, follow these steps to restore service.

4.1 Log Forensic Analysis

Retrieve the state of the system at the time of the incident using the timestamped prober.

# Pull 100 lines of context around a specific timestamp
./scripts/probe-logs.sh all --grep "2026-03-03T10:15" --context 50

4.2 State Reset & Environment Purge (SPEC-53)

If the application state becomes corrupted or environment variables are inconsistent, perform a deep reset from the project root.

  • Soft Reset: mise run reset. Clears item database and local ingest/storage files.
  • Deep Slate Reset: mise run reset --hard.
    • Behavior: Performs a complete environment purge.
    • Actions: Wipes the database, deletes all binary storage, clears all Docker volumes, and removes the local Caddy configuration.
    • Integrity: Enforces root-level routing priority for the main application to resolve landing page conflicts.
  • WARNING: The --hard flag is destructive and non-reversible. It is intended for recovery from total system failure or for starting a clean development slate.

Phase 5: Public Connectivity & OAuth

5.1 Cloudflare Tunnel

To receive eBay callbacks and support OIDC identity providers, route your public domains to your local instance.

  1. Create a tunnel in Cloudflare and add two Public Hostnames mapping to http://localhost:8080:
    • threadscout.yourdomain.com (Main App)
    • auth.yourdomain.com (Identity Provider)
  2. Start the tunnel: mise run tunnel.

5.2 eBay Portal Configuration

Update your Auth Accepted URL in the eBay Developer Portal: https://threadscout.yourdomain.com/api/ebay/callback

5.3 High-Availability Static Compliance (SPEC-48)

To satisfy OIDC registration requirements and ensure high availability, the /privacy and /about pages are deployed as static assets to GitHub Pages.

  • URL: https://docs.threadscout.com/privacy
  • Fallback: This static deployment ensures OIDC compliance even during local backend maintenance, allowing identity providers (Google, eBay) to always access required legal documentation.

5.4. Project-App Static Separation (SPEC-52)

To improve performance and security, ThreadScout separates application binaries from documentation and marketing assets.

  • Frontend Assets: Managed by Next.js and served from the frontend/.next directory.
  • Project Assets: Documentation, ADRs, and analysis reports are processed into static HTML and served via a separate pipeline.
  • Benefit: This separation ensures that application updates do not interfere with documentation availability and reduces the attack surface of the main application.

5.5. Secure Shadow Docs Deployment (SPEC-51)

Internal documentation and pre-release feature specifications are managed via a “Shadow Docs” strategy.

  • Pipeline: These docs are built into a non-publicly indexed directory.
  • Access: Access is restricted to authenticated developers and operations staff via the /ops dashboard.
  • Purpose: Allows the squad to maintain a single source of truth for upcoming features without leaking competitive intelligence to the public landing page.
