Engineering Analysis: Bring Your Own Model (BYOM)

Date: February 28, 2026
Status: Initial Assessment - Concept Design

1. Executive Summary

Following the “Bring Your Own Storage” (BYOS) initiative, this analysis evaluates the feasibility and architectural impact of “Bring Your Own Model” (BYOM). This feature would allow users to provide their own AI API keys (e.g., Google GenAI, OpenAI, Anthropic) or connect to local LLM runners (e.g., Ollama), offloading AI token costs from ThreadScout while giving users control over their data and model selection.

2. Current State

ThreadScout is currently coupled to Google Gemini 2.0/2.5 via the google-genai SDK, running on Vertex AI.

  • Backend Logic: backend/src/threadscout/services/ai.py initializes a global Vertex client.
  • Consumption: research.py uses this client for both analyze_item_images (Vision) and enrich_item_from_url (Text/Search).
  • Configuration: Prompts are stored in config.py and are highly optimized for Gemini’s behavior and JSON output format.

3. The “BYOM” Architectural Gap

Component Current State Required for BYOM
SDK google-genai (Fixed) Generic AIProvider interface.
Identity System-wide ADC / Service Account Per-user encrypted API keys.
Prompts Fixed XML-style (Gemini-optimized) Model-aware prompt templates.
Quotas Managed by ThreadScout (monthly limits) Unlimited/Self-managed by user.
Vision Gemini types.Part.from_bytes Standardized Base64/URL vision handling.

4. Proposed Architecture: AIProvider

We should follow the same pattern as the StorageProvider abstraction.

A. Interface

class AIProvider(ABC):
    @abstractmethod
    async def analyze_images(self, images: List[bytes], prompt: str) -> Dict[str, Any]:
        """Performs vision extraction from images."""
        pass

    @abstractmethod
    async def generate_text(self, prompt: str, system_instruction: str) -> Dict[str, Any]:
        """Performs text-based enrichment/reasoning."""
        pass

B. Supported Backends

  1. GoogleGenAIProvider: (Default) Current implementation using Vertex/SDK.
  2. OpenAIProvider: Compatible with OpenAI (GPT-4o), Groq, and local runners like Ollama or vLLM via the standard /v1/chat/completions endpoint.
  3. AnthropicProvider: Support for Claude 3.5 Sonnet (often preferred for vision/accuracy).

5. Engineering Challenges

A. Prompt Sensitivity (The “JSON” Problem)

ThreadScout relies heavily on structured JSON output. While Gemini 2.5 is robust with response_mime_type="application/json", other models (especially local ones) may require different framing or “schema enforcement” (e.g., Guidance or Outlines).

  • Solution: The AIProvider must handle “Response Format” wrapping for the specific model.

B. Vision Payload Normalization

Gemini handles raw bytes natively in the SDK. OpenAI and Anthropic require Base64 encoding inside a specific message block.

  • Solution: The provider abstraction must take responsibility for payload formatting.

C. Security & Credential Management

Users would provide sensitive API keys.

  • Security: Must be encrypted at-rest using Fernet, identical to the eBay and BYOS Storage tokens.
  • Privacy: User keys never touch logs; usage metadata is still logged locally for the user’s dashboard.

6. Business & UX Impact

Benefits

  • Reseller Autonomy: Pro users can use their own “massive” limits if they are power-users.
  • Cost Offloading: ThreadScout can offer a lower subscription tier if the user provides their own “Compute.”
  • Privacy: Users can point ThreadScout to a local Ollama instance, ensuring photos never leave their local network (requires vision-capable local models like LLaVA or Moondream).

Risks

  • Support Burden: Troubleshooting “Why did the AI fail?” becomes harder when the user is using an obscure local model.
  • Fragmented Accuracy: The core “Draft Quality” metric depends on the model. We would need to define “Recommended Models” (Gemini 2.0 Flash, GPT-4o-mini).

7. Recommendation

Proceed with caution but prioritize the abstraction. Refactoring ai.py into an AIProvider interface is a high-value DX improvement even without BYOM, as it makes the system easier to test.

Next Steps:

  1. Implement AIProvider base class.
  2. Add EncryptedUserSetting support for API keys.
  3. Implement an “OpenAI Compatible” provider to support the widest range of alternatives (Ollama, Groq, GPT).
