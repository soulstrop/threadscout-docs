Architectural Philosophy

ThreadScout is built on four core pillars:

  1. Local-First, Cloud-Augmented: User data and secrets remain local (SQLite/Local Storage), while heavy lifting is offloaded to highly optimized AI models (Gemini) via secure APIs.
  2. Modular Isolation: The backend is strictly separated from the frontend. Operations like AI analysis and eBay pushing are treated as “Pluggable Services,” allowing for easy swaps (e.g., ADR 0003: Pluggable Ingest Backends).
  3. Provable Identity: Every request is scoped to a verified user, even in “Local Mode,” ensuring a seamless transition to multi-user SaaS environments (ADR 0005: Secure Authentication).
  4. Hardware-Level Security: The system is designed to run within restricted environments (Docker/MicroVMs) without sacrificing functionality (ADR 0007: Modular Reverse Proxy).

Developer Experience (DX) & Tooling

ThreadScout prioritize a Zero-Friction developer environment. To achieve this, we utilize a unified toolchain and environment isolation strategy.

  • Unified Orchestration: mise manages the entire stack (run dev, run mock, run test).
  • Environment Isolation: uv for Python and local reverse proxying via Caddy.
  • Secret Authority: Encrypted secrets via fnox.
  • Sandbox: QEMU MicroVMs for isolated AI task execution.

For detailed workflows, setup instructions, and quality standards, see the Developer Guide.

Security & Network Topology

Hybrid Hosting Architecture

ThreadScout utilizes a dual-layered ingress to balance public accessibility (for eBay callbacks) with private execution security.

  • Static Layer (GitHub Pages): Serves public marketing and documentation.
  • Dynamic Layer (Cloudflare Tunnel): Securely routes public traffic to the local application.
  • Documentation Bridge: A specialized build pipeline that ensures browseability across environments. It utilizes:
    • Link Translation: Lua filters to rewrite internal .md links to .html in the generated output.
    • Index Normalization: Ensures INDEX.md is served as index.html at https://localhost:3443/docs and https://threadscout.constans.dev/docs.
    • Auto-indexing: Dynamically generates index pages for subdirectories like adr/ and analysis/.
  • See Also: ADR 0009: Hybrid Hosting Strategy and Design: Hybrid Deployment & Unified Routing.

Mobile Ingress & PWA

  • Android: Utilizes the Web Share Target API.
  • iOS: Uses a custom iOS Shortcut to POST photos to the unified upload endpoint.

Key Patterns & Service Logic

AI Orchestration

AI calls are separated into:

  • Analysis: High-precision vision extraction (Gemini 2.0 Flash).
  • Enrichment: High-speed, lower-cost text processing (Gemini 2.5 Flash-Lite).
  • Compute Abstraction (BYOM): (Planned) The system is transitioning to a pluggable AIProvider architecture. This will enable:
    • Multi-Model Support: Seamless switching between Gemini, OpenAI (GPT-4o), Anthropic (Claude), and local runners like Ollama.
    • Compute Offloading: Users can provide their own API keys to bypass system quotas and manage their own costs.
  • Security: Analysis: LLM Security details our mitigations against prompt injection and resource exhaustion.

eBay Integration Pipeline

We utilize the eBay Inventory/Offer API (modern) rather than the legacy Trading API.

  • Policy Sync: Shipping, Payment, and Return policies are cached locally.
  • Token Management: ADR 0006: Stable Dev OAuth Callback explains how we maintain a stable endpoint for eBay’s redirect.

State Management & Observability

ThreadScout uses a robust state machine to track the lifecycle of every listing.

  • Workflow State: IDLE -> ANALYZING -> SYNTHESIZING -> PUSHING -> SUCCESS.
  • Debug Snapshots: Every state transition triggers a snapshot of the frontend context, persisted to the backend for high-fidelity error reproduction.
  • Logging: We use OTel-compatible structured logging (Structlog) across the entire stack.
  • Operations: See Design: Telemetry & Monitoring.

Multi-tenancy & SaaS Foundations

ThreadScout is designed for secure multi-tenant execution.

  • Isolation: Physical files are siloed at storage/{user_id}/{item_id}/.
  • Identity: We use Auth.js as a broker for multiple providers (Google, Apple, eBay, PocketID).
  • Security: All requests undergo cryptographic JWT verification in the FastAPI layer.
  • Storage Abstraction: The system utilizes a pluggable StorageProvider architecture, allowing for Local Filesystem (Dev) or S3-compatible (SaaS) backends.

Storage & BYOS Credential Setup

Users can “Bring Your Own Storage” (BYOS) by providing their own S3/R2 credentials. These are validated via a “Flight Check” and encrypted at-rest.

  • See Also: Analysis: Multi-tenancy Architecture.

Incident Triage & Monitoring

ThreadScout incorporates a self-contained Ops Pulse system for managing the application’s health and user feedback.

  • Incident Capture: Users can report bugs, performance issues, or accuracy concerns via a sidebar modal.
  • High-Fidelity Context: Every incident report is automatically linked to the current Debug Snapshot, capturing the full state of the frontend and AI context at the time of the report.
  • Deep Health Checks: The system performs tiered health monitoring (Liveness, Readiness, and Deep). The Deep Health Check (/api/health) verifies connectivity to downstream dependencies like Gemini, SQLite, and the OIDC Identity Provider (PocketID).
  • Sentry Integration: All unhandled exceptions and performance bottlenecks are traced via Sentry, with events linked back to the user’s Incident report when applicable.
  • Metrics: A Prometheus endpoint (/api/metrics) provides granular telemetry on AI token usage, storage growth, and eBay draft success rates.

Environment & Execution Matrix

The application’s behavior is determined by its Identity Posture and Dependency Mode.

  • mocking: Uses MockEBayService and MockVertexService (No network egress).
  • development: Real APIs, verbose logging.
  • production: Real APIs, hardened security, structured logging.
