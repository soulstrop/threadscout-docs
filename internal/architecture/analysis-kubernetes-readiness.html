Engineering Analysis: Kubernetes & Observability Readiness

Date: February 15, 2026 Status: Assessment Complete - Migration Planned

Current State Assessment

1. Kubernetes Compatibility

  • Probes: The FastAPI backend lacks dedicated /health, /ready, or /live endpoints. Kubernetes currently relies on generic API endpoints which may not accurately reflect system health (e.g., database connectivity or filesystem write access).
  • Containerization: No Dockerfile exists for either the frontend or backend.
  • Configuration: Application configuration is handled via environment variables and Pydantic Settings, which is K8s-friendly, but no manifests (deployment.yaml, service.yaml) are present.

2. Observability (Prometheus/Grafana)

  • Metrics: The application is not instrumented with a Prometheus exporter. There is no /metrics endpoint.
  • Tracing: While structlog is configured for OTel-compatible JSON logging, there is no distributed tracing integration (e.g., OpenTelemetry SDK).
  • Custom Metrics: Token usage is tracked in SQLite but not exposed as Prometheus gauges/counters, preventing real-time alerting on AI costs.

Technical Recommendations

Phase 6 Implementation Plan

  1. Health Module: Add a threadscout.api.health module to verify SQLite connectivity and storage/ permissions.

  2. Prometheus Middleware: Integrate starlette-prometheus to capture standard HTTP metrics.

  3. Custom Exporters: Implement a background task to export TokenUsage stats as Prometheus metrics.

  4. Multi-stage Dockerfiles:

    • Backend: Use ghcr.io/astral-sh/uv for fast, reproducible builds.
    • Frontend: Use a Node.js build stage followed by a lightweight Nginx or Node runner stage.
