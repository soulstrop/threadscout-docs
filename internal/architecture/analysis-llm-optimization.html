Engineering Analysis: LLM Optimization and Pipeline Cost-Efficiency

Date: 2026-03-12 Status: Under Consideration Authors: Architecture Squad

1. Executive Summary

ThreadScout relies heavily on Large Language Models (LLMs) to automate the translation of raw product photography into structured, highly converting eBay listings. While the current pipeline is functional, the core challenge of any AI-augmented pipeline is the inherent tension between Quality/Accuracy and Cost/Latency per Item.

This document analyzes ThreadScout’s current use of LLMs, explores optimization levers available within the ingestion pipeline, and evaluates long-term strategic investments, such as fine-tuning, model distillation, and self-hosting. Finally, it proposes a rigorous testing framework to empiricaly evaluate these architectural decisions.

2. Current Architecture & LLM Usage

ThreadScout currently utilizes a two-tiered AI strategy via managed cloud providers:

  1. Vision & Extraction (Gemini 2.0 Flash): Processes pre-processed images (resized and background-removed) to identify brand, size, material composition, and condition. This is the most computationally expensive and token-heavy operation.
  2. Synthesis & Enrichment (Gemini 2.5 Flash-Lite): Handles text-based processing. This includes taking raw OCR data, cross-referencing extracted details with external eBay Comp data, generating SEO-optimized titles, and writing human-readable descriptions.

The Ingestion Pipeline

  1. Upload: Mobile client uploads images.
  2. Pre-processing (Deterministic): Resizing (down to 1536px), rotation correction, and U-Net-based background removal for all gallery images.
  3. Hero Image Generation: One processed image is sent to a high-level generative model (e.g., Nano Banana) to synthesize a “worn on translucent mannequin” primary visualization.
  4. Vision Extraction: Gallery Images + Structured Prompt -> Gemini 2.0 Flash -> JSON Payload.
  5. Enrichment: Extracted JSON + Web Scrape Data -> Gemini 2.5 Flash-Lite -> Enhanced Listing Details.
  6. Draft Creation: Push structured data to eBay Inventory API.

3. The Optimization Levers (Cost vs. Quality)

To achieve the best quality at the minimum cost per item, we have several architectural “levers” we can pull dynamically:

3.1 Input Resolution and Image Count

  • Lever: The resolution of images sent to the Vision model, and the number of images required.
  • Trade-off: High resolution (e.g., native 12MP) and multiple angles improve OCR accuracy (especially for tiny tags) and condition spotting, but exponentially increase input token costs and latency.
  • Current State: Resizing to max 1536px. We can optimize further by sending only the cropped region of interest (e.g., just the tag) to the Vision model instead of the whole garment, drastically reducing tokens.

3.2 Generative Visualizations (The “Hero” Image)

  • Lever: Utilizing advanced image generation models (like Nano Banana) to synthesize a high-CTR “translucent mannequin” visualization for the primary search result.
  • Trade-off: Image generation APIs are typically priced per image rather than per token, and high-fidelity generation can be relatively expensive compared to text/vision tasks. It also adds 10-20 seconds of latency to the pipeline.
  • Strategy: This cost must be tracked entirely separately from the OCR/Reasoning token costs. It should be treated as a premium “Value Add” lever. If user margins are low, we can dynamically disable the Hero generation and fall back to the standard white-background image.

3.3 Model Tiering and Routing

  • Lever: Choosing which model handles which task (e.g., Flash vs. Pro vs. Flash-Lite).
  • Trade-off: Smaller models (Flash-Lite) are extremely cheap and fast but may hallucinate details or fail on complex unstructured text. Larger models (Pro) are accurate but economically unviable for low-margin clothing items.
  • Strategy: Implement a “Cascade” approach. Attempt extraction with the cheapest model first. If confidence is low or schema validation fails, escalate to a more expensive, capable model.

3.3 Prompt Engineering and Context Length

  • Lever: The size and complexity of the prompt and few-shot examples.
  • Trade-off: Providing 10 examples of perfectly formatted listings improves output stability but consumes input tokens on every request.
  • Strategy: Utilize prompt caching (where supported by the provider) for static system instructions to drastically reduce the amortized cost of input tokens.

4. Strategic Investments: Specialized Assets

As volume scales, relying solely on generic foundation models may become suboptimal.

4.1 Fine-Tuning and Post-Training

  • Opportunity: The task of converting clothing tags and physical traits into structured JSON is a highly specific, repetitive domain.
  • Cost/Benefit: Fine-tuning a smaller model (like an 8B parameter model) on tens of thousands of ThreadScout’s successful extractions can result in a model that matches the accuracy of a massive foundation model (like GPT-4o or Gemini 1.5 Pro) but at a fraction of the inference cost and latency. It also reduces the need for massive prompts, saving input tokens.

4.2 Distillation and Sparse Models

  • Opportunity: Creating a specialized sparse model (e.g., using Mixture of Experts) or distilling a fundamental model down to a highly constrained schema-extraction engine.
  • Cost/Benefit: High upfront R&D cost to create the dataset and perform the distillation. However, the runtime inference cost becomes near zero compared to API calls. This is only viable if the domain (clothing resale) remains relatively static and the volume is massive (millions of items/month).

4.3 Guardrails and Deterministic Fallbacks

  • Opportunity: Investing in rigid programmatic guardrails (e.g., Pydantic schema validation, brand dictionaries, fuzzy matching against eBay’s specific category trees).
  • Cost/Benefit: High ROI. Instead of asking the LLM to strictly adhere to an eBay category ID, ask it for a textual description and use a fast, deterministic local search (like Elasticsearch or Typesense) to map it to the exact ID. This offloads work from the expensive LLM to cheap traditional compute.

5. Cloud APIs vs. Self-Hosting

At what scale does it make sense to abandon managed providers (Google, OpenAI) and host our own weights on raw cloud compute (AWS EC2, RunPod)?

  • Managed Providers (Current): Best for variable workloads, low operational overhead, and access to state-of-the-art vision models. Pricing is per-token.
  • Self-Hosting (e.g., vLLM on A100s): Pricing is per-hour regardless of token volume.
  • The Tipping Point: Self-hosting only makes economic sense when:
    1. The system has enough sustained, concurrent volume to keep the GPU utilization near 100% 24/7. Spiky, unpredictable workloads will result in paying for idle compute.
    2. We have successfully fine-tuned a smaller model (e.g., Llama 3 8B or Gemma 2 9B) that can perform the task accurately. Hosting a 70B+ parameter model is computationally prohibitive for this use case.
  • Verdict: ThreadScout should remain on managed providers, utilizing the BYOM (Bring Your Own Model) architecture, until sustained volume exceeds the cost of a dedicated GPU instance (~$1,500 - $3,000/month per node) and a fine-tuned small model is proven to work.

6. Testing System for Evaluating Vague Trade-offs

Because questions about “quality vs. cost” are vague, we must implement a rigorous, empirical evaluation harness before changing production routing.

6.1 The “Golden Dataset”

We must curate a dataset of ~1,000 diverse, challenging real-world items (blurry tags, weird brands, vintage pieces) with human-verified “perfect” JSON outputs.

6.2 The Evaluation Harness (CI/CD for LLMs)

We will build a testing script that runs the Golden Dataset through proposed pipeline changes and measures:

  1. Accuracy (F1 Score): Did it get the brand right? Did it miss a flaw? (Evaluated programmatically via strict matching or via “LLM-as-a-Judge”).
  2. Cost per Item: Total tokens consumed * provider pricing.
  3. Latency: Time from image ingest to JSON output.

6.3 A/B Routing (Shadow Mode)

Before fully committing to a new model or prompt, we route 10% of production traffic through the new pipeline in “shadow mode.” We compare its output to the primary pipeline without affecting the actual user draft, allowing us to catch edge-case hallucinations in the wild.

Appendix: Future/Alternative Uses of LLMs in ThreadScout

Beyond the core ingestion pipeline, LLMs could be leveraged for:

  1. Conversational Support (Chatbot): Helping users debug eBay API connection issues or explaining why an item was rejected by eBay’s policy.
  2. Dynamic Pricing Strategy: Analyzing historical eBay sold comps (unstructured text) and generating a narrative recommendation for pricing (e.g., “Price higher because this vintage tag is highly sought after”).
  3. Trend Forecasting: Scraping fashion blogs or TikTok metadata and generating weekly reports for resellers on what brands/styles to source.
  4. Auto-Replies for Buyers: Drafting polite, contextual responses to incoming eBay buyer messages (e.g., “Can you measure the inseam?”) by referencing the item’s stored metadata.
  5. Multi-Lingual Localization: Automatically translating the final listing into multiple languages for cross-border trade, adapting idioms appropriately.
