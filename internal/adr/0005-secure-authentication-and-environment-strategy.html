ADR 0005: Secure Authentication and Environment Strategy

Status

Proposed

Context

ThreadScout is transitioning from a purely local tool to a “Cloud Native Local” application that integrates with external identity providers (eBay, Google, Apple). This introduces strict security requirements for OAuth redirects and public internet exposure, which conflict with our traditional localhost development environment.

We need a unified strategy for:

  1. OAuth Compliance: eBay and others strictly require public https:// callbacks and reject localhost.
  2. Identity Management: Future support for “Login with Google/Apple” will impose similar redirect URI restrictions.
  3. Security: Exposing a local development machine to the public internet (via tunnels) creates a significant attack surface.
  4. Environment Parity: Minimizing “it works on my machine” issues when moving between Dev, CI (GitHub Actions), and Production (likely containerized).

Issues & Considerations

1. The OAuth “Localhost” Problem

  • Constraint: Identity providers (IdPs) like eBay do not consider http://localhost secure or valid for callbacks in production-tier apps.

  • Current Workaround: Using cloudflared to create a temporary public tunnel (https://random-name.trycloudflare.com).

  • Risks:

    • Tunnel Stability: URLs change every restart, requiring constant reconfiguration of the IdP portal.
    • Security: Anyone with the URL can hit your local backend API. There is no WAF or authentication in front of the tunnel by default.

2. Multi-Provider Identity Needs

  • Google/Apple Sign-In: These providers require verifying domain ownership. We cannot verify trycloudflare.com.
  • Requirement: We need a stable, verifiable domain name (e.g., dev.threadscout.app) that points to the developer’s local environment or a staging instance.

3. Production & CI Parity

  • Containerization: Production will be Docker/Kubernetes based. Developing directly on the host OS (mise run dev) creates drift (e.g., different system libraries, permission models).
  • CI Environment: GitHub Actions runners are ephemeral and headless. They cannot perform interactive OAuth flows involving browser redirects. Testing strictly needs to be mocked or run against a long-lived staging environment.

4. Developer Machine Security

  • Risk: Running a public tunnel directly to a workstation bypasses the router’s NAT/Firewall. If the application has a vulnerability (e.g., RCE), the developer’s personal machine is compromised.

  • Mitigation Options:

    • Isolated VM/Container: Run the entire stack (Backend + Frontend + Tunnel) inside a VM or Dev Container. If compromised, the blast radius is contained.
    • Authenticated Tunnel: Use Cloudflare Access or ngrok with OAuth protection on the tunnel itself. (Challenge: The eBay webhook cannot authenticate to the tunnel).

Proposed Strategy

A. Development Environment (The “Sandboxed Tunnel”)

  1. Architecture: Shift development to a Dev Container or Local VM.

    • The mise environment, database, and source code live inside the isolated boundary.
    • The Cloudflare Tunnel runs inside this container.

  2. Stable Domains: Register a domain (e.g., threadscout-dev.com).

    • Use cloudflared with a named tunnel to route https://mike.threadscout-dev.com to the local container.
    • Configure eBay/Google/Apple to trust strictly https://mike.threadscout-dev.com.

B. CI/CD Pipeline

  1. Mocking First: Continue to rely heavily on the mocked E2E tests (workflow.spec.ts) for logic verification.
  2. Contract Testing: Use contract tests (e.g., Pact) to verify that our mocks match the real eBay/Google API behavior, rather than hitting live APIs in CI.

C. Production Transition

  1. Container First: The Dockerfile used for the Dev Container should be nearly identical to the Production image.
  2. Orchestration: Use docker-compose.yaml to define the relationship between Frontend, Backend, Caddy, and the Tunnel. This file becomes the “source of truth” for how the system runs anywhere.

Decision Drivers

  • Safety: We must assume the development endpoint is hostile territory if it is public. Isolation is non-negotiable.
  • Stability: Changing callback URLs daily is unsustainable. We need fixed URLs.
  • Compliance: We must meet the strict HTTPS requirements of our partners.
