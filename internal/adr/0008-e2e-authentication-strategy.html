ADR 0008: E2E Authentication Strategy (Robotic Login)

Status

Proposed

Context

The introduction of authentication middleware (NextAuth.js) and JWT verification in Phase 7 has caused a regression in the Playwright E2E test suite. Tests that previously accessed the dashboard directly are now redirected to the /login page.

We need a stable, high-fidelity method for authenticating tests that:

  1. Verifies the end-to-end integration of the auth middleware.
  2. Does not rely on brittle cookie injection.
  3. Works in headless CI environments.

Decision

We will adopt a “Robotic Login” strategy for E2E tests.

Instead of bypassing authentication, Playwright test scripts will:

  1. Navigate to the /login page.
  2. Click the “Developer Login (Mock)” button (which is active in the mocking environment).
  3. Verify the redirect back to the dashboard before proceeding with the main test logic.

Long-term Recommendation

While “Robotic Login” is sufficient for current development, we recommend transitioning to an Identity Provider Sidecar (using a dedicated instance of Pocket ID for testing) in the production-readiness phase. This will allow for testing real OIDC handshakes and “Bring Your Own Identity” scenarios with 100% environment parity.

Options Considered

1. Middleware Bypass (Negative Constraints)

  • Pros: Fastest test execution.
  • Cons: Creates a security “blind spot” where middleware misconfigurations are not caught by tests.

2. Cookie Injection

  • Pros: Avoids UI interaction.
  • Cons: Brittle; requires internal knowledge of NextAuth session encryption/hashing which frequently changes between versions.

3. Robotic UI Login (Selected)

  • Pros: High fidelity; tests the real user experience; leverages existing “Mock Mode” infrastructure.
  • Cons: Adds minimal overhead (~500ms) to the test setup phase.

Consequences

  • Test Fidelity: Tests now explicitly cover the login and session establishment flow.
  • Maintenance: The E2E beforeEach hook must be updated to handle the login interaction.
  • Security: Ensures that the mocking environment’s developer bypass is correctly restricted and functional.
