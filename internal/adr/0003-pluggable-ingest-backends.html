ADR 003: Pluggable Ingest Backends

Status

Proposed

Context

ThreadScout originally assumed all photos would be located in a local ingest/ directory. However, resellers use various sources for their product photography, including Google Photos, Apple iCloud, and direct URLs.

We need a way to support these varied sources without hardcoding specific logic for every provider into the core application.

Decision

We will implement an Ingest Provider Interface.

The backend will be refactored to use a strategy pattern. A base IngestProvider class will define the contract, and specific implementations (e.g., LocalProvider, GooglePhotosProvider) will handle the platform-specific logic for listing, fetching, and finalizing images.

Interface Contract

Every Provider must implement:

  1. list_images(): Returns metadata for available images (id, preview_url).
  2. get_image_bytes(image_id): Returns raw bytes for AI analysis.
  3. finalize(image_ids, destination_path): Moves or marks images as processed.

Rationale

  • Extensibility: New backends (S3, Dropbox, Apple Photos) can be added by simply implementing the interface.
  • Decoupling: The core FastAPI logic won’t need to know if an image is on a local disk or a Google server.
  • Mobile Workflow: Supporting cloud providers like Google Photos enables a seamless mobile-to-desktop workflow for users.

Consequences

  • Positive: High flexibility; supports modern cloud workflows.
  • Negative: Adds abstraction complexity; cloud providers may require complex OAuth or scraping logic.
  • Migration: The existing local filesystem logic must be moved into a LocalProvider implementation.
