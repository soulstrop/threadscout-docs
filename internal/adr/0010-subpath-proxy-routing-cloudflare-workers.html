0010 Subpath Proxy Routing Cloudflare Workers

ADR 0010: Subpath Proxy Routing via Cloudflare Workers

Status

Proposed

Context

Following the decision in ADR 0009 to adopt a hybrid hosting strategy, we need a way to serve both static content (GitHub Pages) and dynamic content (Cloudflare Tunnel) under a single, unified domain: threadscout.constans.dev.

A standard DNS CNAME can only point to one target. To achieve subpath-based routing (e.g., /docs to GitHub and /api to the Tunnel) on the same domain, we require an interceptor at the edge.

Decision

We will use a Cloudflare Worker as a “Traffic Cop” at the edge of the threadscout.constans.dev domain.

  1. Primary Domain: threadscout.constans.dev will have its DNS CNAME pointed to GitHub Pages ([username].github.io).
  2. Origin Subdomain: The Cloudflare Tunnel will be reconfigured to a private origin subdomain: origin-threadscout.constans.dev. This subdomain is used strictly as a back-end target and is not intended for direct user access.
  3. Routing Logic:
    • Static (Pass-through): Requests for /, /docs/*, and static assets (images/css) will pass through to the primary CNAME target (GitHub Pages).
    • Dynamic (Proxy): Requests for /api/*, /login/*, /auth/*, and /dashboard/* will be proxied by the Worker to the origin-threadscout.constans.dev target.

Consequences

  • Unified Branding: Users never see a subdomain; the transition from documentation to the application is seamless.
  • Edge Efficiency: Routing decisions happen at the CDN level (Cloudflare Edge), minimizing latency.
  • Complexity: Requires maintaining a small JavaScript Worker script and configuring Worker Routes in the Cloudflare Dashboard.
  • Free Tier Usage: Cloudflare Workers offer a generous free tier (100k requests/day) which is more than sufficient for the current scale.
