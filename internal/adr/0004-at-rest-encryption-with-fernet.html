ADR 0004: At-Rest Encryption for Sensitive Tokens via Fernet

Status

Accepted

Context

ThreadScout stores dynamic, user-specific credentials (eBay OAuth2 Access and Refresh tokens) in a local SQLite database. While application-level secrets (API keys) are managed via fnox and injected as environment variables, these session tokens change frequently at runtime and must be persisted. Storing them in plain text within the database poses a significant security risk if the local filesystem is compromised.

We evaluated several options for securing these tokens:

  1. Plain Text: No encryption. (Rejected: Unacceptable security posture).
  2. Database-Level Encryption (e.g., SQLCipher): Encrypting the entire SQLite file. (Rejected: Increases build complexity and requires specialized binaries).
  3. Application-Level AES-GCM: Manual implementation of Authenticated Encryption. (Rejected: Higher risk of implementation errors such as IV reuse).
  4. Fernet (Symmetric Encryption): A high-level, “recipes” based implementation of AES-128 in CBC mode with HMAC for authentication.

Decision

We chose Option 4: Fernet Encryption.

We will use the cryptography library’s Fernet implementation to encrypt sensitive strings before they are written to the database and decrypt them upon retrieval. The master encryption key (32-byte URL-safe base64 string) will be managed as a static secret via fnox and injected into the application environment as THREADSCOUT_ENCRYPTION_KEY.

Comparison of Alternatives

Primitive / Recipe Evaluation Reason for Rejection in ThreadScout
AES-GCM High performance; hardware accelerated. Brittle. Requires manual nonce management. Nonce reuse results in catastrophic key/plaintext recovery.
AES-GCM-SIV / AES-SIV Nonce-misuse resistant. Low-level primitive. Requires manual handling of tags/nonces in the database schema.
XChaCha20Poly1305 High software performance; large 192-bit nonce. Low-level primitive. While safer than GCM, it still requires managing the output byte format manually.
Fernet High-level “Recipe” (AES-CBC + HMAC). Selected. Provides a single self-contained, versioned string that includes the IV and authentication tag.

Rationale

  • Abstraction over Primitives: Fernet is an opinionated, high-level standard that handles key derivation, IV generation, and message authentication (HMAC) automatically. This “un-screw-up-able” API prevents common cryptographic pitfalls like nonce reuse or failing to authenticate the ciphertext.
  • Developer Ergonomics: Because Fernet outputs a single URL-safe string, it avoids the need to modify the database schema to store separate Nonces or Tags, which is where implementation bugs often occur.
  • Safety by Design: While primitives like AES-GCM are faster, the performance difference for small OAuth tokens is measured in microseconds and is negligible compared to the benefit of Fernet’s built-in authentication and versioning.
  • Developer Workflow Integration: Since ThreadScout already uses fnox and mise for environment management, storing a single master key in fnox is seamless and requires zero infrastructure changes.
  • Minimal Dependencies: The cryptography library is a standard, well-maintained Python package already present in the project’s dependency tree.
  • Portability: This approach works across any database backend (SQLite, PostgreSQL, etc.) without requiring specialized database drivers or extensions.

Consequences

  • Positive: Tokens are protected at rest; key rotation is possible by updating the fnox secret; code remains simple and readable.
  • Negative: If the master key is lost, all existing session tokens become unreadable, requiring the user to re-authenticate with eBay.
  • Future Work: Implement a utility for rolling/rotating keys if needed for higher security environments.
