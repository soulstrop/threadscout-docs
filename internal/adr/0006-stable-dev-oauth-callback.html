ADR 0006: Stable Development OAuth Callbacks via Cloudflare

Status

Proposed

Context

ThreadScout integrates with the eBay API, which requires a public HTTPS callback for OAuth2 authentication. Currently, we use temporary Cloudflare Tunnels (trycloudflare.com), which has several drawbacks:

  1. URL Instability: Every time the tunnel restarts, the URL changes. This requires manual updates to the eBay Developer Portal’s “Auth Accepted URL” (RU_NAME).
  2. Compliance: Many IdPs (including eBay) are becoming stricter about redirect URIs, often requiring verified domain ownership or rejecting “random” subdomains.
  3. Local HTTPS Complexity: Running Caddy locally to provide HTTPS for localhost is often blocked by system-level permissions (e.g., binding to port 80/443) or browser certificate trust issues.

Proposed Decision

We will use a stable, custom subdomain (e.g., threadscout.constans.dev) managed by Cloudflare to route traffic to the developer’s local machine.

1. Persistent Cloudflare Tunnel

Instead of a temporary tunnel, we will use a Named Tunnel via cloudflared.

  • The tunnel will be linked to the constans.dev zone.
  • A DNS CNAME record will point threadscout.constans.dev to the tunnel ID.

2. Traffic Routing

  • Traffic arriving at https://threadscout.constans.dev will be proxied by Cloudflare to the local FastAPI backend (e.g., http://localhost:8000).
  • This eliminates the need for a local Caddy proxy for external callbacks.

3. Security Hardening

To ensure this “hop” is safe:

  1. Cloudflare Access (Zero Trust):

    • Protect the root domain with an authentication policy (e.g., Google Workspace, GitHub, or One-Time Pin).
    • Bypass Rule: Create a specific bypass rule for the OAuth callback path (/api/ebay/callback) to allow eBay’s servers to hit the endpoint without manual authentication.

  2. OAuth State Parameter:

    • Implement a cryptographically secure state parameter in the OAuth flow to prevent CSRF attacks. (Implemented in PR #XX).

  3. Strict HTTPS:

    • Enforce “Always Use HTTPS” and HSTS at the Cloudflare edge.

  4. Localhost Binding:

    • The backend should only bind to 127.0.0.1 (not 0.0.0.0) to ensure only the tunnel can reach it.

Alternatives Considered

1. Tailscale Funnel

Tailscale Funnel allows exposing a local port to the public internet via the Tailnet.

  • Pros: Zero-configuration for existing Tailscale users; automatic HTTPS; high security via the Tailnet’s private infrastructure.
  • Cons: Restricted to *.ts.net subdomains (cannot use custom branded domains like constans.dev); all-or-nothing public access to the port (lacks granular path-level bypass rules like Cloudflare Access).

2. Comparison: Cloudflare vs. Tailscale Funnel

Feature Cloudflare Tunnel (cloudflared) Tailscale Funnel
Domain Control Full (threadscout.constans.dev) Restricted (*.ts.net)
Access Control Granular (WAF, Path-level Bypass) Port-level (All or Nothing)
Identity Management Multiple IdPs (Google, GitHub, OKTA) Tailscale Identity (SSO/User)
Branding Professional & Consistent Utility & Infrastructure focused

3. Coexistence Strategy

ThreadScout will utilize both technologies for maximum flexibility and security:

  • Public Ingress (cloudflared): Used strictly for official API callbacks (eBay, Google) that require stable, custom-branded URLs and granular path-level security bypasses.
  • Private Access (Tailscale): Used for secure, authenticated access to the raw developer environment (e.g., accessing the local frontend/backend from a mobile device or laptop on the same Tailnet) without exposing the application to the public internet at all.
  • No Conflict: cloudflared operates at the application proxy layer, while Tailscale operates at the network interface layer. They can run concurrently on the same machine without interference.

Decision Drivers

  • Stability: Reduce developer friction by providing a “set it and forget it” callback URL.
  • Safety: Leverage Cloudflare’s WAF and Zero Trust features to protect the developer machine.
  • Professionalism: Using a real domain is a prerequisite for moving from Sandbox to Production API tiers.
  • Granularity: The ability to bypass Cloudflare Access only for the specific /api/ebay/callback path is critical for external IdP compatibility.

Consequences

  • Dependency: Requires a Cloudflare account and a registered domain.
  • Configuration: Developers must have cloudflared installed and authenticated on their machine.
  • Cost: While the features used are available on the Free Tier, it introduces a reliance on external infrastructure.
