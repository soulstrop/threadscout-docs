SPEC-74: Unified Smoke Test & Profile Verification

1. Executive Summary

This specification establishes a “fail-fast” smoke test suite (check:smoke) for ThreadScout. It verifies that the active profile (mocking, local, hybrid, or live) is correctly provisioned, connected to its required sidecars/external services, and adheres to its specific security and architectural constraints.

2. Profile Architectural Layouts

2.1. mocking Profile

Intent: Fast, offline-first UI/UX development with predictable data.

  • Workstation (Host): Frontend (Next.js), Backend (FastAPI with mock_ebay=True), SQLite (Mocked dataset).
  • Sidecars (Docker): None required.
  • External/Edge:
    • GitHub Pages: Serves static assets/documentation.
    • Cloudflare Worker: Simulated routing/maintenance segment.
  • OAuth Flow: Mocked logic within the application; stubs satisfied locally.

2.2. local Profile

Intent: Full-stack integration testing on the local workstation.

  • Workstation (Host): Frontend, Backend, SQLite (Development DB).
  • Sidecars (Docker/Process): PocketID (OIDC Provider), Caddy (Reverse Proxy for Local HTTPS on :3443).
  • External/Edge:
    • GitHub Pages: Serves static assets/documentation.
    • Cloudflare Worker: Simulated routing/maintenance segment.
  • OAuth Flow: Local PocketID instance satisfying OIDC requirements.

2.3. hybrid Profile

Intent: Local development with public visibility for external webhook/callback verification.

  • Workstation (Host): Frontend, Backend, SQLite, PocketID, Caddy.
  • Sidecars (Process): Cloudflared (Inbound Tunnel).
  • External/Edge:
    • Cloudflare Tunnel: Routes public traffic to the workstation.
    • eBay API: Connected to Sandbox or Production environments.
    • Google/Apple OAuth: Live OIDC providers.
  • OAuth Flow: Live/Sandbox credentials issued to the local application via public URLs.

2.4. live Profile

Intent: Production-parity security and performance verification.

  • Workstation (Host): Hardened Frontend/Backend, SQLite (Encrypted at-rest), PocketID, Caddy (Strict SSL/TLS).
  • Sidecars (Process): Cloudflared (Strict Tunnel).
  • External/Edge:
    • Same as Hybrid, but with strict production credentials and locked-down endpoints.
  • Security Constraint: debug=False, non-default secret_key, and strict JWT verification.

2.5. Edge Routing Logic (The “Appropriate” Flow)

The “Outside World” perspective (Edge) must route traffic based on path and session state:

  • Root (/):
    • Logged In -> Workstation (App).
    • Logged Out -> Static Asset Host (GitHub Pages: Landing Page).
  • Documentation (/docs/*):
    • Always -> Static Asset Host (GitHub Pages).
  • Compliance (/privacy, /about):
    • Always -> Static Asset Host (GitHub Pages).
  • API/Auth (/api/*, /login, /_next):
    • Always -> Workstation (App).

3. Implementation Requirements

3.1. Command Orchestration

  • Task: check:smoke in root mise.toml.
  • Script: scripts/smoke-test.sh (Bash).
  • Dependencies: The script should be runnable independently or via mise.

3.2. Verification Logic (The “Smoke” Checks)

The implementation must verify the following layers based on the active FNOX_PROFILE:

A. Environment Handshake

  • Verify THREADSCOUT_APP_ENV matches the intended profile.
  • Verify THREADSCOUT_INGRESS_MODE (DIRECT vs PROXY vs TUNNEL) is consistent with the profile definition.

B. Service Connectivity (Internal)

  • Backend: GET /api/health returns 200 OK and valid JSON with database: ok.
  • Frontend: GET /api/health-check returns 200 OK and valid JSON.

C. Edge Routing Verification (The “Outside World” Perspective)

Verify the routing logic through the active “Edge” entry point (Wrangler, Caddy, or App):

  • Root Path (/):
    • In live/hybrid, ensure it returns the “Appropriate” content (Landing Page if unauthenticated).
    • In mocking, ensure it returns the App/Dashboard (as it is DIRECT).
  • Documentation Path (/docs/):
    • Verify it returns “ThreadScout Documentation” from the intended source.
    • Differentiate between local docs/gen (Caddy) and remote GitHub Pages (Worker).

D. Identity & OAuth

  • OIDC Provider: Verify the .well-known/openid-configuration endpoint is responsive.
  • Auth Mode: Ensure THREADSCOUT_ENABLE_AUTH is correctly set.

E. Production Hardening (Live Profile Only)

  • Fail if THREADSCOUT_DEBUG is true.
  • Fail if THREADSCOUT_SECRET_KEY is a development/mock string.
  • Fail if THREADSCOUT_ENCRYPTION_KEY is missing.
  • Verify THREADSCOUT_ENABLE_AUTH is true.

F. Edge Segment Consistency

  • Verify reachability of threadscout.constans.dev (if configured).
  • Verify that the hostname maps to 127.0.0.1 locally to prevent production leak.
