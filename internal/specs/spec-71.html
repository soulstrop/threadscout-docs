SPEC-71: C2PA Content Credentials Integration for Image Provenance

1. Goal

Implement the Coalition for Content Provenance and Authenticity (C2PA) standard to cryptographically sign and embed provenance metadata into photographs processed by ThreadScout. This ensures that any AI-driven manipulations (e.g., background removal, enhancement, or resizing) are transparently recorded, maintaining trust in the reseller’s product images.

This spec enables Journey [UJ-02] AI Extraction and addresses Responsible AI practices for image manipulation.

2. Background

ThreadScout ingests raw product photography and leverages AI (Gemini) for extraction and potential image enhancements. To adhere to modern content authenticity standards, any image modified or generated by the system must include a C2PA manifest. This manifest acts as a “nutrition label” for the image, detailing its origin and any edits applied.

3. Implementation Strategy

We will use the official c2pa-python SDK (which provides Python bindings to the Rust c2pa-rs library) to read, modify, and sign manifests during the image processing pipeline.

Prerequisites & Configuration

  • Certificates: C2PA signing requires an X.509 certificate and a private key.
    • For local development (dev profile), the system will generate or use a self-signed certificate.
    • For production (hybrid/prod profiles), the system expects a valid certificate issued by a recognized Certificate Authority (CA), injected via environment variables or file paths managed by fnox.

Processing Pipeline

  1. Ingestion: When a user uploads a photo, the system reads any existing C2PA manifest to preserve previous provenance.
  2. Manipulation: As the image passes through the pipeline (e.g., AI enhancement, cropping), we accumulate a list of C2PA Action assertions (e.g., c2pa.edited, c2pa.resized).
  3. Signing: Before saving the final image to the siloed storage (storage/{user_id}/{item_id}/), the accumulated manifest is signed using the ThreadScout private key and embedded into the image file.

Backend

  • backend/src/threadscout/services/provenance.py (New File):
    • Encapsulate the c2pa-python logic.

  • backend/src/threadscout/services/image_processor.py (Existing/To Be Modified):
    • Inject the provenance service. Ensure that after every Pillow (PIL) or OpenCV manipulation, the provenance service is called to append the corresponding C2PA action before the final save.

    C2PA_PRIVATE_KEY_PATH: Path | None = Field(default=None, description="Path to PEM private key")
    C2PA_CERT_CHAIN_PATH: Path | None = Field(default=None, description="Path to PEM cert chain")
    C2PA_ISSUER_NAME: str = Field(default="ThreadScout System", description="Name of the software agent")

Assertions Schema (Internal mapping)

We will map internal actions to standard C2PA action URIs:

  • c2pa.edited: General AI enhancement.
  • c2pa.resized: Scaling for thumbnails or standardization.
  • c2pa.cropped: Aspect ratio changes.
  • c2pa.removed.background: Specific action for background removal tools.

7. Error Handling & Failsafes

  • Graceful Degradation: If C2PA_ENABLED is false, or if the certificates are missing/invalid, the system should log a warning but must not fail the image processing pipeline. The image should be saved normally without the manifest.
  • Format Constraints: C2PA is primarily supported on JPEG and PNG formats. If the system processes unsupported formats (e.g., obscure WEBP variations where SDK support might be limited), the provenance service should catch the exception, log it, and gracefully return the unsigned image.
  • Performance: C2PA signing adds a small amount of I/O overhead. This should be executed asynchronously or in a background worker thread so it does not block the main FastAPI request thread returning the UI response.
