SPEC-66: Domain-Driven Model Splitting

1. Goal

Refactor the monolithic models.py into a structured package organized by business domain. This eliminates the “god file” anti-pattern and makes it easier to find and modify data structures.

2. Traceability

  • Phase: Phase 8: Production Readiness (DX).
  • Objective: Improve code discoverability and reduce merge conflicts in a multi-contributor environment.

3. Target Files

  • backend/src/threadscout/models/__init__.py: (New) Export all models for backward compatibility.
  • backend/src/threadscout/models/user.py: (New) User, UserTier.
  • backend/src/threadscout/models/item.py: (New) Item, ItemState.
  • backend/src/threadscout/models/incident.py: (New) Incident, IncidentSeverity.
  • backend/src/threadscout/models/auth.py: (New) OAuthToken, Session.

4. Implementation Strategy

  1. Create Package: Convert models.py into a directory models/.
  2. Logical Split: Distribute classes based on their primary domain.
  3. Circular Dependency Guard: Use the __init__.py or local imports within methods to prevent circular references between related models (e.g., User and Item).
  4. Verification: Ensure all existing imports (from threadscout.models import ...) continue to work via __init__.py exports.
