SPEC-73: Grafana Dashboard for High-Throughput Ingestion Queue

1. Goal

Design an operational Grafana dashboard tailored for ThreadScout’s high-throughput, AI-augmented ingestion pipeline backed by ScyllaDB. The dashboard must provide Ops with immediate visibility into system load, database throughput, application-level pipeline latency, and LLM token costs.

This spec supports the proof-of-concept phase for scaling the data backend.

2. Background

As ThreadScout scales to handle 100s of TPS (Transactions Per Second) from distributed mobile clients, Ops needs to monitor three distinct layers simultaneously:

  1. Infrastructure/Database Layer: Is ScyllaDB keeping up with the write volume? Are nodes healthy?
  2. Application Pipeline Layer: How long does it take an item to move from “Ingested” to “Pushed to eBay”?
  3. Economics Layer: Are we hitting AI token rate limits, and what is our real-time cost burn?

3. Data Sources (Prometheus & ScyllaDB)

The dashboard will rely on two primary data sources:

  • Prometheus (System/DB Metrics): Scraping ScyllaDB’s native /metrics endpoint and the FastAPI backend’s /metrics endpoint.
  • ScyllaDB (Application Metrics): Occasional aggregate queries via a Grafana Cassandra/Scylla plugin to calculate business-logic latency and token usage.

4. Dashboard Layout & Panels

The dashboard is organized into three horizontal rows (panels):

Row 1: Executive / Economics Summary (The “Wallet”)

  • Panel: Total Items Ingested (24h): Single stat.
    • Metric: Count of items where status = 'ingested' (from DB) or threadscout_items_ingested_total (from Prometheus).
  • Panel: Token Burn Rate (Tokens/Min): Time-series graph.
    • Metric: Rate of threadscout_llm_tokens_total over a 1m window. Split by model (e.g., gemini-2.0-flash vs gemini-2.5-flash-lite).
  • Panel: Cost per Item (Estimated): Gauge.
    • Metric: (Total Token Cost in last hour) / (Total Items Processed in last hour). Gives immediate feedback if a prompt change caused a cost regression.

Row 2: Pipeline Latency (The “Queue”)

  • Panel: End-to-End Latency (Ingest -> Push): Histogram / Heatmap.
    • Metric: Application-level metric calculating pushed_at - ingested_at. Shows the P50, P90, and P99 latency. If P99 spikes, the LLM or eBay API is likely stalling.
  • Panel: Stage Bottlenecks: Bar chart.
    • Metric: Average time spent in each stage:
      • vision_latency = enriched_at - ingested_at
      • ebay_latency = pushed_at - enriched_at
  • Panel: Items in “Pending” State: Single stat (Red/Yellow/Green thresholds).
    • Metric: Count of items currently stuck without reaching status = 'pushed'. Acts as an alert if the queue backs up.

Row 3: ScyllaDB Health (The “Engine”)

  • Panel: Database Throughput (Read/Write OPS): Time-series graph.
    • PromQL: sum(rate(scylla_storage_proxy_coordinator_read_operations_total[1m])) and write_operations_total.
  • Panel: Coordinator Latency (P99): Time-series graph.
    • PromQL: ScyllaDB 99th percentile write latency. Ensures the database isn’t the bottleneck for mobile uploads.
  • Panel: Node Load / CPU Utilization: Gauge per node.
    • PromQL: sum(rate(node_cpu_seconds_total{mode!="idle"}[1m])) to ensure the cluster is evenly balanced.

5. Implementation Strategy (Future)

  1. Prometheus Integration: Ensure the ThreadScout FastAPI backend exposes business metrics (using the prometheus_client library defined in SPEC-71) to track threadscout_items_ingested_total and threadscout_llm_tokens_total.
  2. ScyllaDB Monitoring Stack: Deploy the standard scylla-monitoring Docker compose stack (which includes Prometheus and Grafana pre-configured with Scylla dashboards).
  3. Custom Provisioning: Write this specific ThreadScout layout as a JSON definition file in infra/grafana/dashboards/threadscout-ops.json so it is automatically provisioned when the monitoring stack boots.

6. Alerting Thresholds

  • High P99 Latency: If End-to-End latency exceeds 45 seconds, trigger a webhook to the Ops channel.
  • Token Spike: If Token Burn Rate jumps 50% above the 7-day moving average, alert for potential looping/retry bugs in the AI services.
  • Dead Letter Queue: If “Items in Pending State” > 100 for more than 5 minutes, alert that the worker queue has crashed.
