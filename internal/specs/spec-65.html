SPEC-65: Configuration & Telemetry Separation

1. Goal

Decouple the static application settings from the dynamic telemetry, logging, and PII-redaction logic. This refactoring restores config.py to a clean Pydantic Settings manifest and centralizes observability logic in a dedicated package.

2. Traceability

  • Phase: Phase 8: Production Readiness (DX).
  • Objective: Reduce cognitive load by separating “What is configured” (Settings) from “How we observe” (Telemetry).
  • User Journey: [UJ-08] Operations & Incident Triage (by clarifying log-scrubbing boundaries).

3. Target Files

  • backend/src/threadscout/config.py: Remove logging and redaction logic; keep only Settings and IngressMode.
  • backend/src/threadscout/telemetry/logging.py: (New) Host for redact_pii_processor and structlog configuration.
  • backend/src/threadscout/telemetry/sentry.py: (New) Host for Sentry initialization and scrubbing callbacks.

4. Implementation Strategy

  1. Extract Redaction: Move redact_pii_processor and its associated regex/whitelists to telemetry/logging.py.
  2. Centralize Logging Configuration: Create a setup_logging(settings: Settings) function in the new package.
  3. Update Main: Import and call the new telemetry setup in the application bootstrap phase.

5. Error Handling

  • Ensure that if telemetry setup fails (e.g., invalid Sentry DSN), the application still boots with basic logging.basicConfig to ensure errors are captured.
