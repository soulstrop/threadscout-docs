Technical Design Spec: Incident Reporting & Triage System (Ops Pulse)

This spec defines the implementation of a full-stack incident reporting, bug tracking, and triage system for ThreadScout.

1. Traceability

  • Goal: Capture and manage application feedback, performance issues, and bugs with high-fidelity context.
  • Journey: [UJ-04: System Administration] (Internal/Ops).
  • Phase: Phase 8: Production Readiness.

2. Terminology & Access Control

Term Description Access Route
Incident A single report (Bug, Slowness, Inaccuracy) created by a user. User (Create) /api/incidents
User Settings User-specific configuration (eBay, Ingest, Theme). User (Manage) /admin
Triage Dashboard Global view of all Incidents across the system for developers. Developer /ops

3. Data Models (SQLModel)

A. Incident Model (backend/src/threadscout/models.py)

from enum import Enum
from uuid import UUID, uuid4
from datetime import datetime
from typing import Optional
from sqlmodel import SQLModel, Field

class IncidentType(str, Enum):
    BUG = "bug"
    PERFORMANCE = "performance"
    ACCURACY = "accuracy"
    SUGGESTION = "suggestion"

class IncidentStatus(str, Enum):
    OPEN = "open"
    TRIAGED = "triaged"
    FIXED = "fixed"
    WONTFIX = "wontfix"

class Incident(SQLModel, table=True):
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    user_id: str = Field(index=True)
    type: IncidentType = Field(default=IncidentType.BUG)
    description: str
    snapshot_id: Optional[UUID] = Field(default=None, foreign_key="debugsnapshot.id")
    sentry_event_id: Optional[str] = None
    status: IncidentStatus = Field(default=IncidentStatus.OPEN)
    created_at: datetime = Field(default_factory=datetime.utcnow)
    metadata: str = Field(default="{}")  # JSON string: { browser, os, url, latency, contact_email }

4. Implementation Plan

A. User-Facing Reporting (frontend/src/components/ReportIssueModal.tsx)

  1. Trigger: Add a “Report Issue” button to the application sidebar and a “Feedback” link in the AI Analysis Preview.
  2. Form: Modal capturing type (dropdown) and description (textarea).
  3. Context Capture:
    • Extract lastSnapshotId from useListingContext().
    • Capture window.location.href and navigator.userAgent.
    • If unauthenticated, allow an optional contact_email.
    • POST to /api/incidents.

B. Backend API (backend/src/threadscout/api/incidents.py)

  1. POST /api/incidents: Creates a new incident. Handles both authenticated and “Guest” reports (Pre-auth).
  2. Security: Implements rate-limiting (5 per hour per IP) and honeypot protection for the public endpoint.
  3. GET /api/incidents: (Ops Only) Returns a paginated list of incidents, sorted by created_at DESC.
  4. PATCH /api/incidents/{id}: (Ops Only) Updates status or adds internal developer notes.

C. Developer Triage Dashboard (frontend/src/app/ops/page.tsx)

  1. High-Density Table: List all incidents with color-coded status badges and “Guest” indicators.
  2. Snapshot Preview: If an incident has a snapshot_id, provide a link to the “Replay View.”
  3. Replay View: A read-only route (/ops/replay/{snapshot_id}) that hydrates the frontend state from the snapshot for visual bug reproduction.

D. Alerting & Integration

  • Webhook: On POST /api/incidents, the backend sends a structured notification to the Ops channel.
  • Sentry Link: If a sentry_event_id is present, generate a direct link to the Sentry issue.

5. Security & Privacy

  1. Anonymization: The metadata field should exclude sensitive tokens or secrets.
  2. Ops Auth: Access to /ops and the associated GET/PATCH API routes is restricted to authorized developers via is_staff and OPS_TOKEN verification.
