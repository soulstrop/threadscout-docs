E2E Authentication in Hardened Environments

Problem Statement

During our CI/CD and local check pipelines (mise run check), Playwright runs E2E tests to verify the application’s behavior. In the dev profile, we rely on the “Developer Login (Mock)” button provided by the NextAuth configuration to bypass the complexities of real OIDC providers.

However, in hardened profiles (such as hybrid or production), the THREADSCOUT_INGRESS_MODE is set to PROXY or TUNNEL, and NEXT_PUBLIC_APP_ENV may not be mocking. Under these conditions, the mock login provider is intentionally omitted from the UI for security reasons.

This causes the auth.setup.ts script in Playwright to time out and fail, as it waits for a “Developer Login (Mock)” button that never renders, thereby failing the entire test suite.

Analysis of Alternatives

To test hardened environments effectively without relying on the mock provider UI, we evaluated the following approaches:

1. Programmatic Session Injection (Recommended)

Instead of interacting with the frontend UI to log in, Playwright bypasses the frontend entirely by generating a valid session behind the scenes.

  • Mechanism: A dedicated, test-only API endpoint (protected by a secure token like THREADSCOUT_BYPASS_BETA_TOKEN) is used to issue a valid JWT or NextAuth session cookie. Playwright fetches this cookie, sets it in the browser context via page.context().addCookies(), and navigates directly to the dashboard.
  • Pros: Highly reliable, extremely fast, completely decoupled from the UI, and works universally across all profiles without exposing a mock provider to end users.
  • Cons: Requires maintaining a dedicated testing endpoint in the backend/Next.js routes.

2. Dedicated “Local/Test” OIDC Provider

Spinning up a localized Identity Provider (e.g., a test instance of PocketID, Keycloak, or Dex) via Docker during the E2E check.

  • Mechanism: NextAuth points to this local provider. Playwright clicks the standard “Sign in with…” button, fills in a hardcoded test credential, and completes the OAuth redirect loop.
  • Pros: Tests the actual OAuth redirect and callback logic natively.
  • Cons: Slower, heavier infrastructure overhead for testing, and prone to flakiness during automated typing/redirects.

3. Segregation of Test Suites

Separating tests into “Workflow E2E” (which requires auth) and “Hardened/Security E2E” (which only tests unauthenticated states and security omissions).

  • Mechanism: Run the full E2E suite only in the dev profile. For hybrid/production profiles, run a subset of Playwright tests that assert the absence of the mock provider and verify redirect boundaries.
  • Pros: Aligns testing goals perfectly with the environment’s capabilities.
  • Cons: Does not test the authenticated workflow under hardened infrastructure configurations (like through the Caddy proxy).

Conclusion and Immediate Action

Long-term Strategy: We will adopt Option 1 (Programmatic Session Injection) to enable robust, authenticated E2E testing across all environments. A specification for this implementation has been created.

Short-term Fix: To immediately unblock the check pipeline for hardened environments, the login steps in auth.setup.ts will be conditionally wrapped. If the environment detects non-DIRECT ingress or a non-mocking configuration, it will gracefully skip the mock login sequence.
