Technical Design Spec: Multi-Provider Authentication (Google & eBay OIDC)

This spec defines the implementation of a Multi-Issuer Identity Broker, allowing users to authenticate directly via Google, eBay, and Pocket ID.

1. Traceability

  • Goal: Enable “Sign in with Google” and “Sign in with eBay” buttons on the login page.
  • Journey: [UJ-00: SaaS Authentication & Onboarding].
  • Phase: Phase 8: Production Readiness.

2. Participants & Responsibilities

Participant Responsibility
Auth.js (Frontend) Orchestrates the OAuth2/OIDC flows for multiple providers.
Google OIDC Identity Provider for accounts.google.com.
eBay OIDC Identity Provider for api.ebay.com.
Pocket ID Local Passkey-only Identity Provider.
FastAPI (Backend) Verifies JWTs from a trusted list of multiple Issuers.

3. Data Models & Configuration

A. Configuration Updates (backend/src/threadscout/config.py)

Add new settings for the additional identity providers:

class Settings(BaseSettings):
    # ... existing settings ...

    # OIDC Providers
    google_client_id: Optional[str] = None
    google_client_secret: Optional[str] = None # Encrypted via Fernet in Prod

    # eBay OIDC (Separate from eBay API credentials)
    ebay_oidc_client_id: Optional[str] = None
    ebay_oidc_client_secret: Optional[str] = None

    # Trust List
    # Map of Issuer URL -> JWKS URL
    oidc_trusted_issuers: Dict[str, str] = {
        "https://accounts.google.com": "https://www.googleapis.com/oauth2/v3/certs",
        "https://localhost:3443/auth": "https://localhost:3443/auth/.well-known/jwks.json",
        "https://api.ebay.com": "https://api.ebay.com/identity/v1/oauth2/jwks"
    }

B. User Identity Mapping

Users will be identified by the pair (iss, sub) to prevent account collisions across providers.

  • User.id: Transformed to provider:internal_id (e.g., google:12345 or pocketid:abcde).

4. Implementation Directives

A. Frontend: Auth.js Configuration (frontend/src/lib/auth.ts)

  1. Add GoogleProvider:

    import GoogleProvider from "next-auth/providers/google";
// ...
GoogleProvider({
  clientId: process.env.THREADSCOUT_GOOGLE_CLIENT_ID,
  clientSecret: process.env.GOOGLE_CLIENT_SECRET,
  checks: ["state"],
})

  2. Add Custom eBay Provider: Since eBay’s OIDC implementation is non-standard, use a custom configuration:

    {
  id: "ebay",
  name: "eBay",
  type: "oauth",
  issuer: "https://api.ebay.com",
  authorization: "https://auth.ebay.com/oauth2/authorize",
  token: "https://api.ebay.com/identity/v1/oauth2/token",
  userinfo: "https://api.ebay.com/identity/v1/user/",
  clientId: process.env.THREADSCOUT_EBAY_OIDC_CLIENT_ID,
  clientSecret: process.env.EBAY_OIDC_CLIENT_SECRET,
  checks: ["state"],
  // ...
}

B. Backend: Multi-Issuer Verification (backend/src/threadscout/auth.py)

  1. Dynamic JWKS Resolver: Update _decode_token to:

    • Decode the JWT header (unverified) to find the iss (Issuer).
    • Check if the iss is in settings.oidc_trusted_issuers.
    • Fetch the corresponding public key from the mapped JWKS URL.
    • Perform full cryptographic verification (jwt.decode) using the correct keys.

  2. JIT Provisioning:

    • Update _jit_provision_user to create the User.id using the iss prefix.

5. Security & Privacy

  1. Issuer Validation: The backend MUST strictly reject any token whose iss claim is not in the explicit oidc_trusted_issuers list.
  2. State Management: NextAuth handles CSRF protection via the state parameter; ensure checks: ["state"] is enabled for all providers.

6. Target Files

  • backend/src/threadscout/config.py: Add provider settings and trust list.
  • backend/src/threadscout/auth.py: Implement Multi-Issuer decoding.
  • frontend/src/lib/auth.ts: Register Google and eBay providers.
  • mise.toml: Map THREADSCOUT_GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, etc. from fnox.
