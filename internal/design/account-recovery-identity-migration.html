Design: Account Recovery & Identity Migration

1. Overview

As ThreadScout moves toward a multi-tenant SaaS model, the durability of user accounts is paramount. Since the application relies on external OpenID Connect (OIDC) providers (Google, PocketID, eBay) for authentication, any loss of access to these providers (e.g., deleted account, lost passkey) could result in an orphaned ThreadScout account.

This design establishes a Multi-Tier Recovery Strategy to ensure users can maintain access to their listings, items, and billing data across identity transitions.

2. Core Objectives

  • Durability: Protect user data from being orphaned due to external identity failures.
  • Flexibility: Allow users to link multiple authentication methods to a single internal account.
  • Security: Prevent account takeovers by enforcing high-trust verification (Email OTP) during the linking process.
  • Auditability: Maintain a forensic record of all administrative identity re-mappings.

3. Architecture: The Multi-Tier Strategy

Tier 1: Multi-Provider Identity Linking (User-Driven)

This is a preventative measure that allows a user to “Self-Heal” their account access.

Workflow:

  1. Detection: A user logs in with a new OIDC provider (e.g., Google) that they haven’t used before on ThreadScout.
  2. Match: The backend detects that the email claim in the new JWT matches an existing verified account.
  3. Prompt: The frontend displays an Identity Linking Modal.
  4. Verification:
    • The backend generates a 6-digit One-Time Password (OTP) and sends it to the verified email on file.
    • The user enters the OTP in the frontend.
  5. Persistence: Upon verification, the new provider:sub pair is added to the user’s identity_mappings list.

Tier 2: Ops-Mediated Identity Re-mapping (Admin-Driven)

This is a last-resort recovery method for cases where OIDC and Email access are both compromised or inaccessible.

Workflow:

  1. Request: User contacts ThreadScout support.
  2. Verification: The administrator verifies the user’s identity through secondary means (e.g., Stripe subscription ID, recent listing details).
  3. Action: The admin uses the Ops Dashboard to manually map a new provider:sub to the user’s internal ID.
  4. Audit: The system logs the change, including the ops_token, the timestamp, and the reason for the re-mapping.

4. Technical Implementation

4.1 Data Model Evolution

The User model transitions from an identity-bound primary key to a collection-based identity model.

User Schema (Conceptual):

{
  "id": "uuid-v4",
  "email": "user@example.com",
  "identity_mappings": [
    {
      "provider": "pocketid",
      "sub": "user_123",
      "linked_at": "2026-03-01T10:00:00Z"
    },
    {
      "provider": "google",
      "sub": "10987654321",
      "linked_at": "2026-03-08T14:30:00Z"
    }
  ]
}

4.2 Frontend State Machine

The Identity Linking flow is managed as a distinct state machine in lib/auth.ts:

  • IDLE -> PROMPT_LINK (on email match)
  • PROMPT_LINK -> OTP_AWAIT (on user consent)
  • OTP_AWAIT -> SUCCESS (on valid code)
  • OTP_AWAIT -> FAILURE (on invalid/expired code)

5. Security Guardrails

  1. OTP Throttling: The backend must rate-limit OTP generation to 3 attempts per hour per account to prevent brute-force linking.
  2. Provider Strictness: Identity linking is only permitted for providers that provide a verified email claim.
  3. Traceability: Every identity mapping change—whether user-driven or admin-driven—must generate an AuditLog entry.
