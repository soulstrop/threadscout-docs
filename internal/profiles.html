ThreadScout: Environment Profiles

This document defines the architectural layout and behavioral constraints for the four supported development and verification profiles in ThreadScout.

2. Profile Definitions

2.1. mocking

Intent: Fast, offline-first UI/UX development.

  • Behavior: The backend runs with mock_ebay=True and satisfies all OIDC requests internally without external providers.
  • Ingress: Browser connects directly to the dev servers on :3000.
  • Use Case: Developing new UI components, testing edge cases with predictable mock data.

2.2. local

Intent: Full-stack integration on the local workstation.

  • Behavior: Uses PocketID (running in Docker) as a real OIDC provider to test the full authentication handshake.
  • Ingress: Access via Caddy reverse proxy on https://localhost:3443 for local HTTPS parity.
  • Use Case: Verifying Auth.js/FastAPI security logic, database migrations, and sidecar connectivity.

2.3. hybrid

Intent: Bridging local development with the public internet.

  • Behavior: Uses Cloudflare Tunnel to expose the workstation to the internet. This is required for verifying eBay OAuth2 callbacks and Webhooks.
  • Ingress: Public URLs (e.g., threadscout.constans.dev) route traffic through the tunnel to Caddy.
  • Use Case: Verifying eBay API integration, Google/Apple OAuth, and real-world network latency.

2.4. live

Intent: Production-parity security verification.

  • Behavior: Enforces strict production constraints defined in config.py:
    • debug=False is mandatory.
    • Default/Mock secret_key and encryption_key are forbidden.
    • Strict JWT verification is enabled.
  • Ingress: High-security tunnel configuration with locked-down endpoints.
  • Use Case: Final pre-release “Red Teaming,” security auditing, and performance benchmarking.

3. Profile Verification (check:smoke)

ThreadScout includes a fail-fast smoke test to ensure your workstation is correctly configured for the selected profile:

# Verify the current profile
mise run check:smoke

This script validates connectivity to the Backend, Frontend, Ingress Proxy, and OIDC providers according to the definitions above.
